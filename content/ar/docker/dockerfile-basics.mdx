# أساسيات Dockerfile: وصفة تطبيقك

تخيّل أنك تتبع وصفة لخبز كعكة. الوصفة تسرد المكونات (صورتك الأساسية)، والخطوات لدمجها (أوامر البناء)، والنتيجة النهائية (صورتك). ملف Dockerfile هو بالضبط ذلك — وصفة تخبر Docker كيف يبني لقطة من تطبيقك، مكتملة بجميع تبعياتها، جاهزة للتشغيل في أي مكان.

في كل مرة تنفذ فيها `docker run nginx`، أنت تستخدم صورة بناها شخص ما باستخدام Dockerfile. في هذا الدرس، ستتعلّم كيف تكتب Dockerfile خاص بك.

## ما هو Dockerfile؟

Dockerfile هو ملف نصي عادي (بدون امتداد — اسمه الحرفي هو `Dockerfile`) يحتوي على سلسلة من التعليمات. كل تعليمة تُنشئ **طبقة** في صورة Docker الناتجة. يقرأ Docker هذه التعليمات من الأعلى إلى الأسفل، وينفّذ كل واحدة، ويحفظ النتيجة كمجموعة مكدسة من الطبقات للقراءة فقط.

فكّر فيه مثل اللازنجيا: كل طبقة تُبنى فوق التي تحتها. الطبقة السفلية هي نظام التشغيل الأساسي، والطبقة التالية تضيف حزم النظام، ثم تبعياتك، وأخيراً كود تطبيقك في الأعلى. إذا غيّرت الطبقة العلوية (كودك)، لن تحتاج إلى إعادة بناء الطبقات السفلية (نظام التشغيل والحزم) — فهي مخزّنة مؤقتاً بالفعل.

## لماذا يهم هذا؟

- **قابلية التكرار** — Dockerfile هو كود. يتم تتبعه بالتحكم في الإصدارات، ومراجعته في طلبات السحب، وينتج نفس الصورة في كل مرة. لا مزيد من "يعمل على جهازي" لأن عملية البناء محددة بالكامل.
- **الأتمتة** — يمكن لخطوط CI/CD بناء صور من ملفات Dockerfile تلقائياً. لا خطوات يدوية، لا أخطاء بشرية.
- **التخزين المؤقت للطبقات يوفّر الوقت** — يخزّن Docker كل طبقة مؤقتاً. إذا لم يتغير شيء في طبقة ما، تُعاد استخدامها. هذا يعني أن إعادة البناء بعد تغيير الكود تستغرق ثوانٍ، لا دقائق.
- **التدقيق الأمني** — يمكنك رؤية ما دخل في الصورة بالضبط. كل حزمة مثبتة، وكل ملف منسوخ موثّق في Dockerfile.
- **التعاون ضمن الفريق** — يمكن لأعضاء الفريق الجدد بناء المشروع بأمر واحد. Dockerfile نفسه هو التوثيق لكيفية إعداد البيئة.

## التعليمات الأساسية، واحدة تلو الأخرى

لنمرّ على كل تعليمة رئيسية في Dockerfile مع مثال عملي.

### FROM — نقطة البداية

```dockerfile
FROM node:20-alpine
```

كل Dockerfile يبدأ بـ `FROM`. تحدد الصورة الأساسية — الأساس الذي تُبنى فوقه كل شيء آخر. فكّر فيه كاختيار نظام التشغيل وبيئة التشغيل التي تريدها مثبتة مسبقاً.

- **`node:20-alpine`** — Node.js 20 على Alpine Linux (توزيعة مصغّرة، حوالي 5 ميغابايت). هذا هو الخيار الأكثر شيوعاً لتطبيقات Node.js.
- **`node:20`** — Node.js 20 على Debian. أكبر حجماً (حوالي 350 ميغابايت) لكنه يتضمن أدوات نظام أكثر.
- **`alpine:3.19`** — Alpine Linux فارغ بدون أي شيء إضافي. تثبّت كل شيء بنفسك.

**لماذا Alpine؟** هي صغيرة، آمنة، وتحتوي على كل ما تحتاجه معظم تطبيقات الخوادم. فضّل متغيرات `-alpine` ما لم تكن تحتاج حزم Debian محددة.

### WORKDIR — تعيين مجلد العمل

```dockerfile
WORKDIR /app
```

يعيّن هذا المجلد الذي ستُنفَّذ فيه جميع الأوامر اللاحقة (RUN, COPY, CMD). إذا لم يكن المجلد موجوداً، ينشئه Docker.

**لماذا لا تستخدم `RUN cd /app`؟** لأن كل أمر `RUN` يعمل في shell جديد. إذا نفّذت `cd` في أمر `RUN` واحد، يبدأ أمر `RUN` التالي من المجلد الأصلي. `WORKDIR` مستمر عبر التعليمات.

### COPY — إحضار كودك

```dockerfile
COPY package*.json ./
```

ينسخ هذا الملفات من جهازك المضيف (سياق البناء) إلى الصورة. الصيغة هي `COPY <مسار_المضيف> <مسار_الصورة>`.

- **`package*.json`** — ينسخ كل من `package.json` و `package-lock.json`. الرمز `*` هو نمط glob.
- **`./`** — ينسخ إلى `WORKDIR` الحالي (`/app`).

هناك أيضاً `ADD`، لكن **فضّل `COPY` لنسخ الملفات البسيط.** `ADD` لديه ميزات إضافية (مثل استخراج ملفات tar وجلب عناوين URL) قد تكون مفاجئة. استخدم `COPY` ما لم تكن تحتاج تحديداً لتلك الميزات.

### RUN — تنفيذ أوامر أثناء البناء

```dockerfile
RUN npm ci --only=production
```

ينفّذ `RUN` أمراً داخل الصورة أثناء عملية البناء. تُحفظ النتيجة كطبقة جديدة.

- **`npm ci`** — "تثبيت نظيف." يثبّت الإصدارات الدقيقة من `package-lock.json`. وهو أسرع وأكثر موثوقية من `npm install` لبيئات CI/البناء.
- **`--only=production`** — يتخطى `devDependencies`. صورتك الإنتاجية لا تحتاج أطر اختبار أو أدوات بناء.

**المبدأ الأساسي:** قلّل عدد طبقات `RUN`. كل `RUN` يُنشئ طبقة جديدة. ادمج الأوامر ذات الصلة:

```dockerfile
# سيئ: 3 طبقات، 3 حاويات وسيطة
RUN apk update
RUN apk add curl
RUN apk add git

# جيد: طبقة واحدة، حاوية وسيطة واحدة
RUN apk update && \
    apk add --no-cache curl git && \
    rm -rf /var/cache/apk/*
```

التنظيف `rm -rf /var/cache/apk/*` مهم — بدونه، يبقى مخزن مدير الحزم في الطبقة إلى الأبد، مما ينتفخ صورتك.

### EXPOSE — توثيق منفذ

```dockerfile
EXPOSE 3000
```

هذا **توثيق، وليس قاعدة جدار حماية.** يخبر أي شخص يقرأ Dockerfile أن التطبيق بداخله يستمع على المنفذ 3000. هو لا ينشر المنفذ فعلياً — لا يزال عليك استخدام `-p 3000:3000` عند `docker run`.

فكّر في `EXPOSE` كتعليق. هو إشارة للمشغّل، وليس فرضاً وقت التشغيل.

### CMD — ما يُشغَّل عند بدء الحاوية

```dockerfile
CMD ["node", "server.js"]
```

يحدد `CMD` الأمر الافتراضي الذي يُنفَّذ عند بدء الحاوية. هناك شكلان مهمان:

- **شكل exec (مفضّل):** `CMD ["node", "server.js"]` — يشغّل `node` مباشرة. لا shell متضمن. معالجة الإشارات تعمل بشكل صحيح (مهمة للإيقاف الآمن).
- **شكل shell:** `CMD node server.js` — يشغّل عبر `/bin/sh -c`. معالجة الإشارات قد تكون معطلة لأن shell هو PID 1، وليس تطبيقك.

**استخدم دائماً شكل exec** ما لم تكن تحتاج تحديداً ميزات shell مثل توسيع المتغيرات.

### الفرق بين CMD و ENTRYPOINT

هذا يربك الكثير من الناس:

| التعليمة | الغرض | هل يمكن تجاوزها وقت التشغيل؟ |
|----------|--------|-------------------------------|
| `CMD` | الأمر الافتراضي أو المعاملات الافتراضية | نعم — `docker run my-app npm start` يستبدل CMD |
| `ENTRYPOINT` | البرنامج التنفيذي الذي يعمل دائماً | أصعب — تحتاج علامة `--entrypoint` |

نمط شائع هو دمجهما معاً:

```dockerfile
ENTRYPOINT ["node"]
CMD ["server.js"]
```

هذا يعني أن `docker run my-app` يشغّل `node server.js`، لكن `docker run my-app worker.js` يشغّل `node worker.js`.

## التخزين المؤقت للطبقات: سر الأداء

هذا أحد أهم المفاهيم في Docker، وهنا يرتكب معظم المبتدئين أخطاءهم.

يبني Docker الصور في طبقات. كل تعليمة تُنشئ طبقة، ويتحقق Docker مما إذا كانت تلك الطبقة قد تغيرت منذ آخر بناء. إذا لم تتغير، يُعيد Docker **استخدام الطبقة المخزّنة مؤقتاً** بدلاً من إعادة بنائها.

**القاعدة الأساسية: رتّب تعليماتك من الأقل تغيراً إلى الأكثر تغيراً.**

```dockerfile
# الطبقة 1: الصورة الأساسية — تتغير تقريباً أبداً
FROM node:20-alpine

# الطبقة 2: مجلد العمل — لا يتغير أبداً
WORKDIR /app

# الطبقة 3: ملفات التبعيات — تتغير عند إضافة/إزالة حزم
COPY package*.json ./

# الطبقة 4: تثبيت التبعيات — تتغير عند تغير package*.json
RUN npm ci --only=production

# الطبقة 5: كود التطبيق — يتغير مع كل التزام
COPY . .

# الطبقة 6: إعدادات وقت التشغيل — نادراً ما تتغير
EXPOSE 3000
CMD ["node", "server.js"]
```

**لماذا ترتيب COPY مهم جداً؟** إذا نفّذت `COPY . .` قبل `npm ci`، ففي كل مرة تغيّر فيها سطراً واحداً من الكود المصدري، تُبطل طبقة `npm ci` أيضاً. هذا يعني إعادة تثبيت جميع تبعياتك من الصفر في كل بناء. بنسخ `package*.json` أولاً وتشغيل `npm ci` قبل نسخ الكود المصدري الكامل، لا تُعاد تثبيت التبعيات إلا عندما تتغير ملفات الحزم فعلياً.

**ما الذي يُبطل التخزين المؤقت؟**

| التغيير | الطبقات المُبطلة |
|---------|------------------|
| تغيير `FROM` | كل شيء بعدها |
| تغيير `package.json` | طبقة `npm ci` وكل شيء بعدها |
| تغيير الكود المصدري | طبقة `COPY . .` وكل شيء بعدها |
| إضافة إلى `.dockerignore` | ربما لا شيء (إذا كانت الملفات المستبعدة هي التي تغيرت) |

## .dockerignore — حافظ على بنائك نظيفاً

يخبر ملف `.dockerignore` Docker أي الملفات يجب استبعادها من سياق البناء. بدونه، يرسل Docker مجلد مشروعك بالكامل إلى البرنامج الخفي — متضمناً `node_modules`، و `.git`، وغيرها من الملفات غير الضرورية.

أنشئ ملف `.dockerignore` في نفس المجلد الذي يحتوي على Dockerfile:

```
node_modules
npm-debug.log
.git
.github
.env
.env.local
Dockerfile
docker-compose.yml
.dockerignore
README.md
.vscode
coverage
dist
```

**لماذا هذا مهم:**

- **سرعة البناء** — سياق بناء أصغر يعني نقل أسرع إلى برنامج Docker الخفي.
- **حجم الصورة** — إذا كانت `node_modules` في سياق البناء، فإن `COPY . .` المتسرع سينسخها إلى الصورة، مما يضاعف الحجم.
- **الأمان** — لا تريد ملفات `.env` (التي قد تحتوي على أسرار) بالقرب من صورتك.
- **كفاءة التخزين المؤقت** — التغييرات في الملفات المستبعدة لن تُبطل تخزينك المؤقت.

## شرح سياق البناء

عندما تنفذ `docker build -t my-app .`، تلك النقطة `.` في النهاية هي سياق البناء. إنه المجلد الذي يرسله Docker إلى البرنامج الخفي. جميع مسارات `COPY` و `ADD` نسبية إلى هذا المجلد.

```bash
# البناء من المجلد الحالي
docker build -t my-app .

# البناء من مجلد محدد
docker build -t my-app /path/to/project

# البناء باستخدام Dockerfile في موقع مختلف
docker build -t my-app -f docker/Dockerfile .
```

**خطأ شائع:** تشغيل `docker build` من جذر مستودع أحادي كبير. قد يكون سياق البناء غيغابايتات. شغّل البناء دائماً من أصغر مجلد يحتوي على ما تحتاجه.

## مثال عملي: بناء تطبيق Node.js

لنبنِ Dockerfile كاملاً وجاهزاً للإنتاج لتطبيق Node.js Express.

**هيكل المشروع:**

```
my-express-app/
├── Dockerfile
├── .dockerignore
├── package.json
├── package-lock.json
├── server.js
└── src/
    ├── routes.js
    └── middleware.js
```

**package.json:**

```json
{
  "name": "my-express-app",
  "version": "1.0.0",
  "main": "server.js",
  "scripts": {
    "start": "node server.js"
  },
  "dependencies": {
    "express": "^4.18.2"
  }
}
```

**server.js:**

```javascript
const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.json({ message: 'Hello from Docker!' });
});

app.get('/health', (req, res) => {
  res.json({ status: 'healthy' });
});

app.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

**Dockerfile:**

```dockerfile
# Use the official Node.js image on Alpine for minimal size
FROM node:20-alpine

# Set the working directory inside the container
WORKDIR /app

# Copy dependency manifests first (for layer caching)
COPY package*.json ./

# Install ONLY production dependencies
RUN npm ci --only=production

# Copy the rest of the application source code
COPY . .

# Document that the app listens on port 3000
EXPOSE 3000

# Run as a non-root user for security
USER node

# Start the application
CMD ["node", "server.js"]
```

**.dockerignore:**

```
node_modules
npm-debug.log
.git
.env
Dockerfile
README.md
```

**البناء والتشغيل:**

```bash
# Build the image
docker build -t my-express-app:1.0 .

# Run the container
docker run -d -p 3000:3000 --name my-app my-express-app:1.0

# Test it
curl http://localhost:3000
# Expected: {"message":"Hello from Docker!"}

# Check health endpoint
curl http://localhost:3000/health
# Expected: {"status":"healthy"}
```

**مخرجات البناء المتوقعة:**

```
[+] Building 12.3s
 => [1/6] FROM node:20-alpine
 => [2/6] WORKDIR /app
 => [3/6] COPY package*.json ./
 => [4/6] RUN npm ci --only=production
 => [5/6] COPY . .
 => [6/6] CMD ["node", "server.js"]
 => exporting to image
 => => writing image sha256:a1b2c3d4...
```

لاحظ كيف تتوافق كل خطوة مع تعليمة في Dockerfile. في بنائك الثاني (مع تغيير الكود المصدري فقط)، تُخزّن الخطوات 1-4 مؤقتاً ويُكمل البناء في ثوانٍ.

## نظرة على البناء متعدد المراحل

ستتعلّم هذا بعمق في درس لاحق، لكن إليك معاينة. البناء متعدد المراحل يتيح لك استخدام مرحلة واحدة للبناء (مع المترجمات وأدوات التطوير) ومرحلة منفصلة مصغّرة للتشغيل. هذا يقلل حجم الصورة النهائية بشكل كبير.

```dockerfile
# Stage 1: Build (has all the heavy tools)
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Run (only what's needed for production)
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

السر الأساسي: `COPY --from=builder` يسحب فقط الملفات التي تحتاجها من مرحلة البناء. كل شيء آخر (المترجمات، الكود المصدري، تبعيات التطوير) يُتجاهل.

## أخطاء شائعة ونصائح

- **تثبيت تبعيات التطوير في الإنتاج** — استخدم دائماً `npm ci --only=production` (أو `--omit=dev`). تبعيات التطوير قد تتضمن أدوات تزيد سطح الهجوم.

- **عدم استخدام `.dockerignore`** — نسيان هذا الملف يعني أن `.git` و `node_modules` وملفات `.env` تُرسل إلى برنامج Docker الخفي. هذا يبطئ البناء وقد يسرّب أسراراً.

- **وضع `COPY . .` قبل تثبيت التبعيات** — هذا أشهر خطأ في Dockerfile. يكسر التخزين المؤقت للطبقات ويجبر على إعادة تثبيت التبعيات مع كل تغيير في الكود.

- **استخدام `ADD` بدلاً من `COPY`** — `ADD` يمكنه جلب عناوين URL واستخراج الأرشيفات، وهو سلوك مفاجئ. استخدم `COPY` لنسخ الملفات البسيط.

- **التشغيل كجذر** — بشكل افتراضي، تعمل الحاويات كجذر. أضف `USER node` (لصور Node.js) أو أنشئ مستخدماً مخصصاً. سنتناول هذا بتفصيل أكبر في درس الأمان.

- **عدم تثبيت علامات الصور** — `FROM node:latest` هدف متحرك. استخدم إصدارات محددة مثل `FROM node:20.11-alpine3.19` لبناءات قابلة للتكرار.

- **عدد كبير من الطبقات** — كل `RUN` و `COPY` و `ADD` يُنشئ طبقة. ادمج أوامر `RUN` ذات الصلة بـ `&&` لتقليل الطبقات وحجم الصورة.
