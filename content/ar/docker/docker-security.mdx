# أمان Docker: إغلاق حاوياتك بإحكام

تخيّل أنك مالك مبنى سكني. كل مستأجر يحصل على وحدته الخاصة بقفل على الباب — هذا عزل الحاويات. لكن جميع المستأجرين يتشاركون أساس المبنى نفسه، والسباكة نفسها، والنظام الكهربائي نفسه — هذه النواة المشتركة. إذا اكتشف مستأجر كيفية الوصول إلى البنية التحتية المشتركة، تكون كل وحدة معرضة للخطر.

هذه هي الحقيقة الأساسية لأمن الحاويات. توفر الحاويات العزل، لكنه ليس نفس الجدار الصلب بين الآلات الافتراضية. استغلال حاوية يصل إلى النواة يمكن أن يؤثر على المضيف بأكمله — وكل حاوية أخرى تعمل عليه. الأمان في Docker ليس تلقائياً؛ بل يتطلب تهيئة متعمدة في كل طبقة.

## خطر النواة المشتركة

لنكن دقيقين بشأن التهديد. تشترك الحاويات في نواة Linux الخاصة بالمضيف. هذا يعني:

- ثغرة في نواة المضيف تؤثر على جميع الحاويات
- الهروب من الحاوية (الوصول إلى المضيف) ممكن إذا كانت النواة تحتوي أخطاء
- الحاويات على نفس المضيف يمكن أن تتداخل مع بعضها عبر موارد النواة

هذا لا يعني أن الحاويات غير آمنة بطبيعتها — بل يعني أنك تحتاج تطبيق الأمن بعمق. لا تعتمد على قفل واحد؛ استخدم طبقات متعددة.

## لماذا يهم هذا؟

- **الحاويات غالباً تعمل في الإنتاج ببيانات حساسة** — بيانات اعتماد قواعد البيانات، مفاتيح API، وبيانات المستخدمين تتدفق عبر حاوياتك. إعداد خاطئ واحد يمكن أن يكشف كل شيء.
- **الإعداد الافتراضي غير آمن** — إعدادات Docker الافتراضية تعطي الأولوية لسهولة الاستخدام على الأمان. تعمل الحاويات كجذر، نظام الملفات قابل للكتابة، ولا توجد حدود للموارد. يجب تأمينها بنشاط.
- **هجمات سلسلة التوريد حقيقية** — الصور الأساسية الخبيثة، التبعيات المخترقة، والتلاعب بالصور هي نواقل هجوم موثقة. تحتاج استراتيجية للتحقق مما تشغله.
- **استغلالات النواة تعبر حدود الحاويات** — على عكس الآلات الافتراضية، استغلال النواة في حاوية واحدة يمكن أن يؤثر على المضيف بأكمله. تقليل الامتيازات يحد من نطاق الضرر.
- **متطلبات الامتثال** — SOC 2 و PCI-DSS و HIPAA جميعها تتطلب ضوابط أمنية محددة. تشغيل حاويات بدون تحصين يعد انتهاكاً للامتثال.

## مكدس أمن الحاويات

فكّر في أمن الحاويات كمكدس من أربع طبقات. تحتاج حماية في كل واحدة:

| الطبقة | ما هو المعرض للخطر | الدفاعات الرئيسية |
|--------|-------------------|-------------------|
| **الصورة** | ثغرات، أسرار، سطح هجوم منتفخ | الفحص، `.dockerignore`، الصور بدون توزيعة |
| **وقت التشغيل** | تصعيد الامتيازات، الكتابة غير المصرح بها | مستخدم غير جذر، نظام ملفات للقراءة فقط، صلاحيات |
| **الشبكة** | الحركة الجانبية، الوصول غير المصرح به | شبكات مخصصة، قيود المنافذ، حركة مشفرة |
| **المضيف** | استغلالات النواة، استنزاف الموارد | Seccomp، AppArmor، حدود الموارد، تحديثات النواة |

لنمرّ على كل دفاع بالتفصيل.

## أقل امتياز: القاعدة الذهبية

مبدأ أقل امتياز بسيط: امنح الحاوية فقط الصلاحيات التي تحتاجها للعمل بالضبط. لا شيء أكثر. إذا كان خادم الويب لا يحتاج تعديل الملفات، فلا تسمح بذلك. إذا لم يحتاج وصولاً خاماً للشبكة، فلا تعطه إياه.

### التشغيل كمستخدم غير جذر

بشكل افتراضي، تعمل حاويات Docker كجذر (UID 0). هذا يعني أن المهاجم الذي يخترق حاويتك يحصل على صلاحيات الجذر داخلها. مع ثغرة في النواة، هذا يمكن أن يؤدي للاستيلاء الكامل على المضيف.

**أصلحه في Dockerfile:**

```dockerfile
FROM node:20-alpine
WORKDIR /app

# Copy files and set ownership to the 'node' user
COPY --chown=node:node . .

# Switch to the non-root user
USER node

CMD ["node", "server.js"]
```

الصورة الأساسية `node` تتضمن مستخدم `node` معد مسبقاً (UID 1000). علامة `--chown=node:node` تضمن أن الملفات مملوكة لهذا المستخدم، و `USER node` يبدّل هوية وقت التشغيل.

**أصلحه وقت التشغيل:**

```bash
docker run --user 1000:1000 my-app
```

هذا يتجاوز توجيه USER في Dockerfile بـ UID:GID محدد. مفيد عندما لا يمكنك تعديل الصورة.

**لماذا يهم هذا؟** العملية التي تعمل كمستخدم غير جذر داخل الحاوية لديها صلاحيات أقل بكثير. لا تستطيع تعديل ملفات النظام، أو تثبيت حزم، أو تغيير إعدادات الشبكة. حتى لو اخترقت الحاوية، تكون خيارات المهاجم محدودة بشدة.

### إسقاط صلاحيات Linux

صلاحيات Linux تُفكّك السلطة الأحادية للجذر إلى أذونات محددة. بشكل افتراضي، تحصل حاويات Docker على مجموعة محدودة من الصلاحيات، لكن يمكنك تقليلها أكثر:

```bash
docker run \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  my-web-app
```

- **`--cap-drop ALL`** — إزالة جميع الصلاحيات أولاً
- **`--cap-add NET_BIND_SERVICE`** — إضافة فقط القدرة على الربط بمنافذ أقل من 1024

هذا هو النهج الأكثر أماناً: ابدأ من لا شيء، أضف فقط ما تحتاجه.

**صلاحيات شائعة ومتى تحتاجها:**

| الصلاحية | ما تسمح به | متى تحتاجها |
|----------|------------|-------------|
| `NET_BIND_SERVICE` | الربط بمنافذ أقل من 1024 | خوادم الويب على المنفذ 80/443 |
| `SYS_PTRACE` | تتبع استدعاءات نظام العملية | أدوات التصحيح (لا تستخدم في الإنتاج) |
| `CHOWN` | تغيير ملكية الملفات | نادراً ما تحتاج وقت التشغيل |
| `SETUID`/`SETGID` | تبديل هوية المستخدم/المجموعة | تكاد لا تحتاج أبداً |
| `DAC_OVERRIDE` | تجاوز فحوص أذونات الملفات | تكاد لا تحتاج أبداً |

### نظام ملفات للقراءة فقط

إذا كان تطبيقك يقرأ الملفات فقط وقت التشغيل (وهو صحيح لمعظم خدمات الويب)، اجعل نظام الملفات للقراءة فقط:

```bash
docker run --read-only --tmpfs /tmp my-app
```

- **`--read-only`** — يركّب نظام ملفات الحاوية بالكامل كقراءة فقط. لا تستطيع الحاوية الكتابة في أي مكان.
- **`--tmpfs /tmp`** — ينشئ تركيبة tmpfs قابلة للكتابة عند `/tmp`. بعض التطبيقات تحتاج مجلداً مؤقتاً لعمليات وقت التشغيل.

**لماذا يساعد هذا:** إذا حاول مهاجم كتابة نص خبيث أو تعديل ملف إعدادات، تفشل الكتابة. نظام الملفات ثابت وقت التشغيل.

**بعض التطبيقات تحتاج مسارات قابلة للكتابة محددة.** يمكنك تركيب tmpfs لها:

```bash
docker run \
  --read-only \
  --tmpfs /tmp \
  --tmpfs /var/run \
  --tmpfs /home/app/.cache \
  my-app
```

## فحص الصور: اعرف ما بداخل صورتك

كل صورة تسحبها أو تبنيها تحتوي برمجيات، والبرمجيات فيها ثغرات. فحص الصور يتحقق من كل حزمة في صورتك مقابل قواعد بيانات الثغرات المعروفة (CVEs).

### الفحص باستخدام Trivy

Trivy هو فاحص مفتوح المصدر شائع وسهل الاستخدام:

```bash
# Install Trivy (macOS)
brew install trivy

# Scan a local image
trivy image my-app:latest

# Scan with severity filtering (only critical/high)
trivy image --severity CRITICAL,HIGH my-app:latest

# Scan and fail the build if vulnerabilities are found
trivy image --exit-code 1 --severity CRITICAL my-app:latest
```

**مثال على المخرجات:**

```
my-app:latest (alpine 3.19.2)
==============================
Total: 2 (CRITICAL: 1, HIGH: 1)

┌───────────┬─────────────┬──────────┬───────────────────┐
│ Library   │ Vulnerability│ Severity │ Fixed Version     │
├───────────┼─────────────┼──────────┼───────────────────┤
│ libcurl   │ CVE-2024-NNN│ CRITICAL │ 8.5.0-r1          │
│ openssl   │ CVE-2024-NNN│ HIGH     │ 3.2.1-r0          │
└───────────┴─────────────┴──────────┴───────────────────┘
```

### الفحص باستخدام Docker Scout

Docker Scout هو أداة الفحص المدمجة في Docker (متاحة مع اشتراك Docker Pro):

```bash
# Scan an image
docker scout cves my-app:latest

# Compare two versions to find new vulnerabilities
docker scout compare --from my-app:1.0 --to my-app:2.0
```

### دمج الفحص في CI/CD

يجب أن يحدث فحص الصور تلقائياً في خط البناء:

```yaml
# GitHub Actions example
- name: Build image
  run: docker build -t my-app:${{ github.sha }} .

- name: Scan for vulnerabilities
  run: |
    trivy image --exit-code 1 \
      --severity CRITICAL,HIGH \
      --ignore-unfixed \
      my-app:${{ github.sha }}
```

هذا يضمن عدم وصول أي صورة بها ثغرات حرجة غير مُصلحة إلى الإنتاج.

## إدارة الأسرار: لا تضع أسراراً في الصور أبداً

هذا أشهر خطأ أمني في Docker: خبز الأسرار (كلمات المرور، مفاتيح API، الرموز) في الصورة. بمجرد أن تحتوي صورة على سر، أي شخص لديه وصول للصورة يمكنه استخراجه — حتى لو حذفت السر في طبقة لاحقة، الطبقة السابقة لا تزال تحتويه.

**ما يُحسب كـسر؟**

- كلمات مرور قواعد البيانات
- مفاتيح API (AWS، Stripe، SendGrid، إلخ)
- مفاتيح توقيع JWT
- مفاتيح TLS الخاصة
- أسرار عميل OAuth

**ما لا يجب فعله:**

```dockerfile
# NEVER DO THIS
FROM node:20-alpine
ENV DATABASE_PASSWORD=hunter2
ENV AWS_SECRET_ACCESS_KEY=AKIA...
```

حتى لو أزلت ENV في طبقة لاحقة، الطبقة الأصلية لا تزال تحتوي القيمة. أي شخص ينفذ `docker history` أو يفحص الطبقات يمكنه رؤيتها.

**ما يجب فعله بدلاً من ذلك:**

### الخيار 1: متغيرات البيئة وقت التشغيل

```bash
docker run -e DATABASE_PASSWORD=hunter2 my-app
```

السر يُمرر وقت التشغيل، ولا يُخبز في الصورة. يوجد فقط في ذاكرة الحاوية.

### الخيار 2: Docker Secrets (وضع Swarm)

```bash
# Create a secret
echo "hunter2" | docker secret create db_password -

# Use it in a service
docker service create \
  --name my-app \
  --secret db_password \
  my-app
```

السر يُركَّب كملف عند `/run/secrets/db_password` داخل الحاوية. مشفر أثناء السكون والنقل، ولا يُكتب في الصورة أبداً.

### الخيار 3: ملفات مركبة وقت التشغيل

```bash
docker run \
  -v /path/to/secrets/db_password:/run/secrets/db_password:ro \
  my-app
```

ركّب ملفاً من المضيف (أو من مدير أسرار) كقراءة فقط. يقرأه التطبيق عند البدء.

### الخيار 4: مديري أسرار خارجيين

للإنتاج، استخدم مدير أسرار مخصص:

- **HashiCorp Vault** — أسرار مركزية مع تدوير وتدقيق
- **AWS Secrets Manager** — متكامل مع IAM وتدوير تلقائي
- **Azure Key Vault** — سحابي أصلي مدعوم بـ HSM
- **GCP Secret Manager** — حل أسرار Google Cloud

تسحب حاويتك الأسرار عند البدء من المدير الخارجي، لإبقائها خارج الصورة بالكامل.

## Docker Content Trust: تحقق من سلامة الصور

كيف تعرف أن الصورة التي سحبتها هي ما قصده الناشر؟ يستخدم Docker Content Trust التوقيعات الرقمية للتحقق من سلامة الصور.

```bash
# Enable content trust for pulls and pushes
export DOCKER_CONTENT_TRUST=1

docker pull my-registry/my-app:1.0
# Docker verifies the image signature before pulling
```

مع تفعيل ثقة المحتوى:
- يمكنك فقط سحب الصور الموقعة
- يمكنك فقط دفع الصور الموقعة
- الصور المُعَدَّلة تُرفض

**ثبّت الصور بالبصمة** لأقوى ضمان:

```bash
# Pull by digest instead of tag
docker pull nginx@sha256:a1b2c3d4e5f6...
```

العلامة (`nginx:1.25`) يمكن أن تشير إلى صور مختلفة بمرور الوقت. البصمة (`sha256:...`) تشير دائماً إلى نفس محتوى الصورة بالضبط. إذا استبدل شخص ما الصورة خلف علامة، لن تتطابق البصمة.

## معايير الأمان: معيار CIS لـ Docker

ينشر مركز أمان الإنترنت (CIS) معيار Docker — قائمة شاملة لأفضل ممارسات الأمان. التوصيات الرئيسية تتضمن:

- أبقِ Docker محدّثاً
- عيّن أذونات نظام ملفات مناسبة لملفات Docker (`/var/lib/docker`، إلخ)
- قيّد وصول شبكة برنامج Docker الخفي
- اضبط قواعد `iptables`
- فعّل إعادة تعيين مساحة أسماء المستخدم
- استخدم `--log-driver` لمنع التلاعب بالسجلات
- عيّن افتراضات `ulimit`
- عطّل التواصل بين الحاويات بشكل افتراضي

يمكنك أتمتة فحوص المعيار باستخدام أدوات مثل **Docker Bench for Security**:

```bash
docker run --rm --net host --pid host \
  --userns host --cap-drop all \
  -v /etc:/etc:ro \
  -v /lib/systemd/system:/lib/systemd/system:ro \
  -v /usr/bin/containerd:/usr/bin/containerd:ro \
  -v /var/lib/docker:/var/lib/docker:ro \
  -v /var/run/docker.sock:/var/run/docker.sock:ro \
  docker/docker-bench-security
```

ينتج هذا تقريراً مفصلاً عن الوضع الأمني لمضيف Docker لديك.

## مثال عملي: Dockerfile إنتاجي محصّن

لنجمع كل شيء في Dockerfile جاهز للإنتاج ومحصّن أمنياً:

```dockerfile
# Multi-stage build for minimal image
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Production stage with security hardening
FROM node:20-alpine

# Install security updates
RUN apk update && apk upgrade --no-cache && rm -rf /var/cache/apk/*

WORKDIR /app

# Copy only production dependencies
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force

# Copy built application from builder
COPY --from=builder /app/dist ./dist

# Create a non-root user and set ownership
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
COPY --chown=appuser:appgroup . .

# Switch to non-root user
USER appuser

# Document the port
EXPOSE 3000

# Health check for runtime monitoring
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD node -e "require('http').get('http://localhost:3000/health', (r) => { process.exit(r.statusCode === 200 ? 0 : 1) })"

# Start the application
CMD ["node", "dist/server.js"]
```

**شغّله بأقصى قيود أمنية:**

```bash
docker run -d \
  --name my-secure-app \
  --read-only \
  --tmpfs /tmp \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  --memory 256m \
  --cpus 0.5 \
  --pids-limit 50 \
  --security-opt no-new-privileges \
  -e DATABASE_PASSWORD="$(cat /run/secrets/db_password)" \
  -p 3000:3000 \
  my-secure-app:1.0
```

لنفكّك كل علامة أمنية:

- **`--read-only`** — نظام الملفات ثابت وقت التشغيل
- **`--tmpfs /tmp`** — فقط `/tmp` قابل للكتابة (في الذاكرة، لا على القرص)
- **`--cap-drop ALL`** — إزالة جميع صلاحيات Linux
- **`--cap-add NET_BIND_SERVICE`** — إضافة فقط القدرة على الربط بالمنافذ المنخفضة
- **`--memory 256m`** — حد الذاكرة 256 ميغابايت (يمنع OOM من التأثير على الحاويات الأخرى)
- **`--cpus 0.5`** — الحد بنصف نواة معالج
- **`--pids-limit 50`** — منع قنابل التشعب (50 عملية كحد أقصى)
- **`--security-opt no-new-privileges`** — منع العملية من الحصول على امتيازات إضافية عبر ملفات setuid/setgid الثنائية
- **`-e DATABASE_PASSWORD=...`** — تمرير الأسرار وقت التشغيل، لا في الصورة

## أخطاء شائعة ونصائح

- **التشغيل كجذر افتراضياً** — هذا أشيع خطأ أمني. أضف دائماً `USER <مستخدم-غير-جذر>` إلى Dockerfile. إذا لم تتضمن الصورة الأساسية مستخدماً غير جذر، أنشئ واحداً.

- **خبز الأسرار في الصور** — متغيرات ENV في Dockerfiles، ملفات `.env` في سياق البناء، وبيانات الاعتماد المضمنة في الكود جميعها مرئية في طبقات الصورة. استخدم أسرار وقت التشغيل بدلاً من ذلك.

- **تجاهل فحص الصور** — الثغرات في الصور الأساسية والتبعيات تُكتشف باستمرار. افحص كل صورة قبل أن تصل إلى الإنتاج، وأعد الفحص وفق جدول زمني.

- **عدم تحديث الصور الأساسية** — صورة Alpine أو Node.js المثبتة منذ ستة أشهر على الأرجح تحتوي ثغرات معروفة. أعد البناء بانتظام للحصول على تصحيحات أمنية.

- **استخدام علامة `--privileged`** — هذا يعطّل جميع القيود الأمنية ويعطي الحاوية وصولاً كاملاً للمضيف. لا يجب استخدامها تقريباً أبداً. إذا ظننت أنك تحتاجها، على الأرجح تحتاج صلاحية محددة بدلاً منها.

- **نسيان `no-new-privileges`** — هذه العلامة الواحدة تمنع العملية من تصعيد الامتيازات. أضفها لكل حاوية لا تحتاج تصعيد امتيازات صراحةً.

- **عدم فحص صور CI/CD** — صور مشغل CI الخاص بك تحتاج فحصاً أيضاً. بيئة بناء مخترقة يمكنها حقن كود خبيث في كل صورة تنتجها.
