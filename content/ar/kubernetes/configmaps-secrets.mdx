# ConfigMaps و Secrets

تخيّل أنك تحزم حقائبك لرحلة. لن تخيط وجهتك على سترتك — بل ستكتبها على بطاقة يمكنك استبدالها. هذا بالضبط المشكلة التي يحلها ConfigMaps و Secrets في Kubernetes. **الإعدادات يجب أن تكون منفصلة عن صورة الحاوية** لكي نفس الصورة تعمل في التطوير، وبيئة التحضير، والإنتاج بإعدادات مختلفة.

بدون هذا الفصل، ستحتاج صورة Docker مختلفة لكل بيئة. غيّرت عنوان قاعدة البيانات؟ أعد بناء الصورة. غيّرت مستوى السجلات؟ أعد البناء مرة أخرى. هذا مُهدِر للوقت، بطيء، وعرضة للأخطاء. الـ ConfigMaps و Secrets تتيح لك حقن الإعدادات وقت التشغيل، مبقيةً صورك نظيفة، قابلة لإعادة الاستخدام، وقابلة للنقل.

## مشكلة الإعدادات

إليك ما يحدث عندما تدمج الإعدادات في الصور:

- **صورة واحدة لكل بيئة.** التطوير، وبيئة التحضير، والإنتاج يحتاج كل منها صورة خاصة بقيم مُضمّنة.
- **تغيير إعداد بسيط يتطلب إعادة بناء كاملة.** تغيير مستوى السجلات لا يجب أن يعني إعادة بناء وإعادة نشر صورة بحجم 500 ميجابايت.
- **الأسرار في الصور خطيرة.** كلمات مرور قواعد البيانات المدمجة في الصور تنتهي في سجل الحاويات. أي شخص لديه وصول للسجل يمتلك بياناتك.
- **لا مرونة.** لا يمكنك تجاوز الإعدادات بدون إعادة البناء. لا يمكنك تجربة إعدادات مختلفة بسرعة.

الحل: **فصل كل شيء يتغيّر بين البيئات.** الـ ConfigMaps تتعامل مع الإعدادات غير الحساسة. الـ Secrets تتعامل مع البيانات الحساسة.

## ما هو الـ ConfigMap؟

الـ **ConfigMap** هو كائن Kubernetes يخزّن بيانات إعدادات غير حساسة كأزواج مفتاح-قيمة. تطبيقك يقرأ هذه البيانات وقت التشغيل عبر متغيرات البيئة أو ملفات محمّلة.

فكّر في الـ ConfigMap كقائمة إعدادات في هاتفك. لا تعيد كتابة نظام تشغيل الهاتف عندما تريد تغيير الخلفية — بل تُحدّث إعدادًا. الـ ConfigMaps تفعل الشيء نفسه لحاوياتك.

### ما يوضع في الـ ConfigMap؟

- أسماء مضيفي قواعد البيانات وأرقام المنافذ
- عناوين URL للخدمات الخارجية
- أعلام الميزات (تفعيل/تعطيل الميزات)
- مستويات السجلات (`info`، `debug`، `warning`)
- مهلات التطبيق وإعدادات إعادة المحاولة
- أي قيمة إعداد غير حساسة

### ما لا يوضع في الـ ConfigMap؟

- كلمات مرور قواعد البيانات (استخدم **Secret**)
- مفاتيح API والرموز (استخدم **Secret**)
- مفاتيح TLS الخاصة (استخدم **Secret**)
- أي بيانات لن تلصقها في قناة Slack عامة

## لماذا يهمّ هذا؟

- **نفس الصورة، بيئات متعددة.** ابن مرة واحدة، انشر في كل مكان. الـ ConfigMaps توفر القيم الخاصة بكل بيئة وقت التشغيل.
- **تغييرات إعداد سريعة.** حدّث ConfigMap وأعد تشغيل الـ Pods — لا حاجة لإعادة البناء. غيّر مستوى السجلات في ثوانٍ، وليس دقائق.
- **بنية مفصولة.** كود تطبيقك لا يحتاج لمعرفة أين يعمل. يقرأ الإعدادات من بيئته فحسب.
- **إدارة إعدادات مركزية.** جميع إعدادات تطبيقك تعيش في مكان واحد، مُتحكم بإصداراتها في YAML، وليس مبعثرة عبر Dockerfiles.
- **التوافق مع منهجية 12-عامل.** الـ ConfigMaps تساعدك على اتباع مبدأ "خزّن الإعدادات في البيئة" من منهجية 12-عامل.

## إنشاء ConfigMaps

هناك ثلاث طرق لإنشاء ConfigMap. لنمضي عبر كل واحدة.

### الطريقة 1: من قيم حرفية

الطريقة الأسرع — حدد أزواج المفتاح-القيمة مباشرة في سطر الأوامر:

```bash
kubectl create configmap app-config \
  --from-literal=DATABASE_HOST=db.production.example.com \
  --from-literal=DATABASE_PORT=5432 \
  --from-literal=LOG_LEVEL=info \
  --from-literal=MAX_CONNECTIONS=100 \
  --from-literal=CACHE_ENABLED=true
```

```text
configmap/app-config created
```

هذا ينشئ ConfigMap بخمسة مفاتيح. رائع للتجارب السريعة والإعدادات البسيطة، لكنه يصبح مرهقًا للإعدادات الكبيرة.

### الطريقة 2: من ملف

لديك ملف إعدادات بالفعل؟ حمّله مباشرة:

```bash
# Creates a ConfigMap with a key named "nginx.conf" containing the file contents
kubectl create configmap nginx-config \
  --from-file=nginx.conf
```

يمكنك أيضًا تحديد اسم مفتاح مخصص:

```bash
kubectl create configmap nginx-config \
  --from-file=my-nginx.conf=nginx.conf
```

هذا مثالي لملفات الإعدادات الكاملة — إعدادات Nginx، application.properties، إعدادات JSON، إلخ.

### الطريقة 3: من ملف YAML بياني (موصى به للإنتاج)

للتحكم في الإصدارات وإمكانية إعادة الإنتاج، استخدم دائمًا ملف YAML بياني:

```yaml
apiVersion: v1                          # Core API group
kind: ConfigMap
metadata:
  name: app-config                      # Name of the ConfigMap
  namespace: production                # Optional -- defaults to "default"
data:                                   # The actual configuration data
  DATABASE_HOST: "db.production.example.com"   # Key-value pairs
  DATABASE_PORT: "5432"                        # All values are strings in ConfigMaps
  LOG_LEVEL: "info"
  MAX_CONNECTIONS: "100"
  CACHE_ENABLED: "true"
  APP_TIMEOUT: "30"                            # Even numbers are stored as strings
```

**مهم:** جميع القيم في الـ ConfigMaps تُخزّن كنصوص. حتى لو كتبت `5432` بدون علامات اقتباس، يعاملها Kubernetes كنص. ضع هذا في اعتبارك عندما يتوقع تطبيقك نوعًا محددًا.

طبّق وتحقق:

```bash
kubectl apply -f app-config.yaml
kubectl get configmap app-config
```

```text
NAME         DATA   AGE
app-config   6      5s
```

عمود `DATA` يعرض عدد المفاتيح التي يحتويها الـ ConfigMap.

```bash
kubectl describe configmap app-config
```

```text
Name:         app-config
Namespace:    default
Data
====
DATABASE_HOST:  db.production.example.com
DATABASE_PORT:  5432
LOG_LEVEL:      info
MAX_CONNECTIONS:100
CACHE_ENABLED:  true
APP_TIMEOUT:    30
```

## استخدام ConfigMaps في الـ Pods

هناك طريقتان لحقن بيانات الـ ConfigMap في حاوياتك: **متغيرات البيئة** و**أحجام محمّلة**. لكل منهما مقايضات مختلفة.

### الخيار 1: متغيرات البيئة (جميع المفاتيح)

النهج الأبسط — حقن كل مفتاح في الـ ConfigMap كمتغير بيئة:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:1.0
      envFrom:                                # Inject ALL keys from the ConfigMap
        - configMapRef:
            name: app-config                  # Reference the ConfigMap by name
```

كل مفتاح في `app-config` يصبح متغير بيئة داخل الحاوية. تطبيقك يقرأ `DATABASE_HOST` و`LOG_LEVEL` وغيرها كأي متغير بيئة آخر.

**المزايا:** بسيط، كل الإعدادات دفعة واحدة.
**العيوب:** كل مفتاح يصبح متغير بيئة، حتى التي قد لا تريدها. لا تحكم دقيق.

### الخيار 2: متغيرات البيئة (مفاتيح محددة)

لتحكم أكبر، أشر إلى مفاتيح فردية:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:1.0
      env:                                    # Define individual environment variables
        - name: DB_HOST                       # The env var name inside the container
          valueFrom:
            configMapKeyRef:
              name: app-config                # Which ConfigMap to read from
              key: DATABASE_HOST              # Which key in the ConfigMap
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
```

**المزايا:** تعرض فقط المفاتيح التي تحتاجها. يمكنك إعادة تسمية المفاتيح (مثلاً، `DATABASE_HOST` يصبح `DB_HOST`).
**العيوب:** YAML أكثر تفصيلًا. تحتاج لسرد كل مفتاح فردي.

### الخيار 3: أحجام محمّلة

لملفات الإعدادات (مثل `nginx.conf` أو `application.yml`)، حمّلها كملفات:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-app
spec:
  containers:
    - name: nginx
      image: nginx:alpine
      volumeMounts:                           # Mount the volume into the container
        - name: config-volume                 # Reference the volume defined below
          mountPath: /etc/nginx/nginx.conf     # Where to mount it inside the container
          subPath: nginx.conf                 # Mount only this specific key (not the whole directory)
          readOnly: true                      # Config files should be read-only
  volumes:                                    # Define the volumes available to the Pod
    - name: config-volume                     # Name must match the volumeMount reference
      configMap:
        name: nginx-config                    # Which ConfigMap to mount
```

بدون `subPath`، يُحمّل الـ ConfigMap بأكمله كدليل، حيث يصبح كل مفتاح ملفًا. مع `subPath`، تحمّل مفتاحًا واحدًا كملف واحد. استخدم `subPath` عندما تريد استبدال ملف محدد واحد، وليس تراكب دليل كامل.

**الفرق الأساسي عن متغيرات البيئة:** عندما يُحمّل الـ ConfigMap كحجم، **التحديثات على الـ ConfigMap تنعكس تلقائيًا في الملفات المحمّلة** (بعد بضع ثوانٍ). متغيرات البيئة تُعيّن وقت إنشاء الـ Pod ولا تُحدّث تلقائيًا.

## ما هو الـ Secret؟

الـ **Secret** يشبه الـ ConfigMap للبيانات الحساسة. يخزّن كلمات المرور، مفاتيح API، شهادات TLS، وبيانات اعتماد أخرى. الفرق الأساسي: الـ Secrets **مشفّرة بـ base64** ويمكن أن تكون **مشفّرة وقت التخزين**.

### تحذير ترميز Base64

هذا يوقع تقريبًا كل مبتدئ: **base64 ليس تشفيرًا.** إنها مجرد خطة ترميز. أي شخص يمكنه فك ترميز سلسلة base64:

```bash
# This is NOT secure:
echo -n "s3cur3P@ss" | base64
# Output: czNjdXIzUEBzcw==

# Anyone can decode it:
echo "czNjdXIzUEBzcw==" | base64 --decode
# Output: s3cur3P@ss
```

ترميز base64 موجود لأن Kubernetes يخزّن البيانات كبايتات ويحتاج طريقة لتمثيل البيانات الثنائية (مثل شهادات TLS) جنبًا إلى جنب مع النص. "أمان" الـ Secrets يأتي من:

1. **RBAC** — يمكنك تقييد من يقرأ الـ Secrets (على عكس الـ ConfigMaps)
2. **التشفير وقت التخزين** — فعّل هذا في إعدادات خادم API الخاص بالكتلة
3. **سجلات التدقيق** — الوصول إلى الـ Secrets يُسجّل
4. **مديري أسرار خارجيين** — ادمج مع AWS Secrets Manager أو HashiCorp Vault أو غيرهما

## إنشاء Secrets

### الطريقة 1: من قيم حرفية

```bash
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=s3cur3P@ss
```

ملاحظة: `kubectl` يتولى ترميز base64 نيابة عنك. تمرر القيمة الخام، وهو يشفرها تلقائيًا.

### الطريقة 2: من ملفات

```bash
kubectl create secret tls app-tls \
  --from-file=tls.crt=server.crt \
  --from-file=tls.key=server.key
```

هذا ينشئ Secret من نوع TLS مخصص للشهادات.

### الطريقة 3: من ملف YAML بياني

عند كتابة YAML مباشرة، **يجب** أن تشفر قيمك بـ base64 بنفسك:

```bash
# Encode the values first
echo -n "admin" | base64
# Output: YWRtaW4=

echo -n "s3cur3P@ss" | base64
# Output: czNjdXIzUEBzcw==
```

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque                            # Opaque = arbitrary key-value data
data:                                   # Values MUST be base64-encoded
  username: YWRtaW4=                    # base64("admin")
  password: czNjdXIzUEBzcw==            # base64("s3cur3P@ss")
```

يمكنك أيضًا استخدام `stringData` بدلًا من `data` لتجنب التشفير اليدوي:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:                             # Raw strings -- Kubernetes encodes them for you
  username: admin
  password: s3cur3P@ss
```

الـ `stringData` للكتابة فقط. عندما تقرأ الـ Secret مرة أخرى بـ `kubectl get`، تظهر القيم مشفرة بـ base64 تحت `data`.

## استخدام Secrets في الـ Pods

### كمتغيرات بيئة (مفاتيح فردية)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:1.0
      env:
        - name: DB_USER                     # Environment variable name
          valueFrom:
            secretKeyRef:                     # Read from a Secret (not ConfigMap)
              name: db-credentials            # Which Secret
              key: username                   # Which key
        - name: DB_PASS
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
```

**أفضل ممارسة:** استخدم `secretKeyRef` للمفاتيح الفردية بدلًا من `envFrom` مع `secretRef`. هذا يتبع مبدأ الامتيازات الأقل — تطبيقك يحصل فقط على الأسرار المحددة التي يحتاجها، وليس كلها.

### كملفات محمّلة

```yaml
volumes:
  - name: secret-volume
    secret:
      secretName: db-credentials             # Mount all keys as files
      defaultMode: 0400                      # Read-only for owner only (octal permission)
volumeMounts:
  - name: secret-volume
    mountPath: /etc/secrets                  # Each key becomes a file: /etc/secrets/username, /etc/secrets/password
    readOnly: true
```

داخل الحاوية، تطبيقك يقرأ `/etc/secrets/username` و`/etc/secrets/password` كملفات عادية. الملفات تحتوي على القيم **المفكوكة** (وليست base64).

## ConfigMap مقابل Secret: مقارنة

| الميزة | ConfigMap | Secret |
|--------|-----------|--------|
| **الغرض** | إعدادات غير حساسة | بيانات حساسة (كلمات مرور، مفاتيح، شهادات) |
| **التخزين** | نص عادي في etcd | مشفر بـ base64 في etcd |
| **حد الحجم** | 1 ميبيبايت لكل ConfigMap | 1 ميبيبايت لكل Secret |
| **RBAC** | وصول عادي للمساحة | وصول أكثر تقييدًا بشكل افتراضي |
| **التشفير وقت التخزين** | لا | اختياري (يجب تفعيله في إعدادات خادم API) |
| **معالجة الذاكرة** | مخزّن في نظام ملفات قائم على القرص | يمكن أن يكون مدعومًا بـ tmpfs (RAM) لأمان إضافي |
| **الإنشاء من قيم حرفية** | نعم | نعم |
| **الإنشاء من ملفات** | نعم | نعم (بما في ذلك نوع TLS) |
| **انتشار التحديثات** | أحجام محمّلة: تلقائي | أحجام محمّلة: تلقائي |
| **صديق لـ Git** | نعم — آمن للإيداع | لا — تجنب إيداع Secrets الخام |

## مثال عملي: تطبيق ويب قابل للإعداد

لننشر تطبيق ويب يقرأ اتصال قاعدة البيانات وإعدادات السجلات من ConfigMap، وبيانات اعتماد قاعدة البيانات من Secret.

**هيكل المشروع:**

```text
k8s/
├── configmap.yaml
├── secret.yaml
├── deployment.yaml
└── service.yaml
```

**configmap.yaml:**

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: web-app-config
data:
  DATABASE_HOST: "postgres.production.svc.cluster.local"   # Internal DNS for the database
  DATABASE_PORT: "5432"                                    # PostgreSQL default port
  DATABASE_NAME: "app_production"                          # Database name
  LOG_LEVEL: "info"                                        # Logging verbosity
  CACHE_TTL: "300"                                         # Cache time-to-live in seconds
  MAX_UPLOAD_SIZE: "10MB"                                  # Maximum file upload size
```

**secret.yaml:**

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: web-app-secrets
type: Opaque
stringData:                                # Using stringData to avoid manual base64
  DATABASE_USER: "app_user"
  DATABASE_PASSWORD: "Ch@ng3M3!NPr0d"
  API_KEY: "sk-prod-abc123xyz789"
```

**deployment.yaml:**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
        - name: web-app
          image: my-registry/web-app:2.0.0
          ports:
            - containerPort: 3000
          envFrom:
            - configMapRef:
                name: web-app-config        # All ConfigMap keys become env vars
          env:
            - name: DATABASE_USER          # Inject specific secret keys
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: DATABASE_USER
            - name: DATABASE_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: DATABASE_PASSWORD
            - name: API_KEY
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: API_KEY
          resources:
            requests:
              cpu: "200m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
```

انشر كل شيء:

```bash
kubectl apply -f k8s/
kubectl get pods -l app=web-app
```

```text
NAME                       READY   STATUS    RESTARTS   AGE
web-app-7d8b9c5f4-2xzkm   1/1     Running   0          30s
web-app-7d8b9c5f4-5njpw   1/1     Running   0          30s
web-app-7d8b9c5f4-8qrty   1/1     Running   0          30s
```

تحقق من متغيرات البيئة داخل Pod:

```bash
kubectl exec -it web-app-7d8b9c5f4-2xzkm -- env | grep -E "DATABASE|LOG_LEVEL|API_KEY"
```

```text
DATABASE_HOST=postgres.production.svc.cluster.local
DATABASE_PORT=5432
DATABASE_NAME=app_production
DATABASE_USER=app_user
DATABASE_PASSWORD=Ch@ng3M3!NPr0d
LOG_LEVEL=info
API_KEY=sk-prod-abc123xyz789
```

تطبيقك يرى هذه كمتغيرات بيئة عادية. ليس لديه فكرة أنها جاءت من Kubernetes — وهذا بالضبط المقصد.

## الأخطاء الشائعة ونصائح

| الخطأ | ما الذي يسوء | كيف تصلحه |
|-------|-------------|-----------|
| **دمج الإعدادات في الصور** | لا يمكن تغيير الإعدادات بدون إعادة البناء | استخدم ConfigMaps و Secrets لجميع القيم الخاصة بالبيئات |
| **تخزين Secrets في Git** | كلمات المرور والمفاتيح تنتهي في التحكم بالإصدارات | استخدم Sealed Secrets أو External Secrets Operator أو SOPS للتشفير قبل الإيداع |
| **نسيان base64 في YAML** | حقل بيانات الـ Secret يحصل على مدخل غير صالح | استخدم `stringData` للقيم الخامة، أو تذكر التشفير بـ `echo -n "value" \| base64` |
| **افتراض أن base64 = تشفير** | الـ Secrets يمكن فكها بسهولة من أي شخص يمكنه قراءتها | فعّل التشفير وقت التخزين؛ استخدم RBAC لتقييد الوصول |
| **عدم تعيين حدود الموارد** | ConfigMap/Secret المحمّل كحجم يمكن أن يستهلك الذاكرة | عيّن `readOnly: true` على أحجام التحميل |
| **تحديث ConfigMaps بدون إعادة تشغيل الـ Pods** | تغييرات متغيرات البيئة لا تُلتقط من الحاويات العاملة | الـ ConfigMaps المحمّلة كأحجام تتحدث تلقائيًا؛ متغيرات البيئة تتطلب إعادة تشغيل الـ Pod |
| **استخدام نفس ConfigMap عبر مساحات الأسماء** | الـ ConfigMaps محدودة بمساحة الأسماء — لا يمكنك الإشارة عبر مساحات الأسماء | أنشئ ConfigMap في كل مساحة أسماء، أو استخدم أدوات تُزامن الـ ConfigMaps |
| **وضع ملفات كبيرة في ConfigMaps** | يصل لحد الـ 1 ميبيبايت | استخدم PersistentVolumes للملفات الكبيرة؛ الـ ConfigMaps لقيم الإعدادات الصغيرة |

### نصائح احترافية

- استخدم `kubectl create configmap --from-env-file=.env` لتحويل ملف `.env` إلى ConfigMap.
- استخدم **Sealed Secrets** (`kubeseal`) إذا كنت بحاجة لتخزين Secrets مشفرة في Git بأمان.
- للإعدادات المؤسسية، فكّر في **External Secrets Operator** لمزامنة الأسرار من AWS Secrets Manager أو HashiCorp Vault أو Azure Key Vault.
- أشر دائمًا إلى مفاتيح أسرار محددة (`secretKeyRef`) بدلًا من استخدام `envFrom` مع الأسرار — هذا يحد من التعرض إذا كان الـ Secret يحتوي على بيانات أكثر مما يحتاجه تطبيقك.
- فعّل **التشفير وقت التخزين** في إعدادات خادم API (`--encryption-provider-config`) للكتل الإنتاجية.
- استخدم `kubectl get secret db-credentials -o jsonpath='{.data.password}' | base64 --decode` لقراءة قيمة Secret بسرعة عند تصحيح الأخطاء.
