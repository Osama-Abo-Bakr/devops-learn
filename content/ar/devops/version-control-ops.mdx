# التحكم بالإصدارات للعمليات

إذا تعطلت إعدادات الخادم في الساعة 3 صباحًا، فالسؤال الأول هو دائمًا نفس الشيء: ما الذي تغير؟ بدون التحكم بالإصدارات، الإجابة هي "قام شخص ما بتسجيل الدخول وتعديل ملف، ولا سجل لما فعله." مع التحكم بالإصدارات، تحصل على سلسلة تدقيق دقيقة: من غيّر ماذا، ومتى، ولماذا (رسالة الالتزام). يمكنك التراجع عن أي تغيير فورًا. يمكنك رؤية التاريخ الكامل لكل ملف تكوين.

التحكم بالإصدارات ليس فقط لكود التطبيق. فرق العمليات تدير التكوينات والبرامج النصية وتعريفات خطوط الأنابيب وبيانات Kubernetes وملفات Terraform وكتيبات التشغيل. كل هذا ينتمي إلى Git. إذا كان شيئًا يحدد كيف تعمل بنيتك التحتية، فيجب تتبعه ومراجعته ونشره من خلال نفس العملية مثل الكود.

## لماذا تحتاج العمليات إلى التحكم بالإصدارات

ثلاثة أسباب تجعل التحكم بالإصدارات ضروريًا للعمليات:

- **سلسلة التدقيق** — كل تغيير له سجل: من أنشأه، ومتى، ولماذا. أثناء الحادث، يمكنك تتبع الالتزام الدقيق الذي أدخل المشكلة والتراجع باستخدام `git revert`.
- **التراجع** — إذا أدى النشر إلى تعطيل الإنتاج، يمكنك العودة إلى أي حالة سابقة بأمر واحد. لا مزيد من إعادة البناء اليدوية من الذاكرة.
- **التعاون** — يمكن لعدة مهندسين العمل على تغييرات البنية التحتية في وقت واحد من خلال الفروع وطلبات السحب، دون التداخل في عمل بعضهم البعض.

بدون التحكم بالإصدارات، يتم إجراء التغييرات مباشرة على الخوادم. لا يوجد تاريخ ولا مراجعة ولا طريقة سهلة للتراجع عن خطأ. هذا هو نمط الخادم الفريد المضاد — كل خادع تكوين فريد وهش لا يمكن لأحد إعادة إنتاجه.

## مراجعة أساسيات Git

إذا كنت جديدًا على Git، فإليك الأوامر التي ستستخدمها يوميًا:

- **`git add`** — إعداد التغييرات للالتزام التالي. يحدد التعديلات التي تريد تضمينها.
- **`git commit`** — تسجيل التغييرات المُعدة برسالة وصفية. هذا ينشئ لقطة دائمة في مستودعك المحلي.
- **`git push`** — إرسال التزاماتك إلى مستودع بعيد (GitHub أو GitLab). هذا يجعل تغييراتك مرئية للفريق.
- **`git pull`** — جلب ودمج التغييرات من المستودع البعيد. افعل هذا قبل بدء عمل جديد لتجنب التعارضات.
- **`git branch`** — إنشاء أو سرد أو التبديل بين الفروع. تعزل الفروع العمل الجاري عن الكود الرئيسي.
- **`git merge`** — دمج التغييرات من فرع إلى آخر. هكذا تعود فروع الميزات عملها إلى الفرع الرئيسي.

سير العمل اليومي:

```
git pull                # جلب آخر التغييرات
git checkout -b fix/timeout-config   # إنشاء فرع ميزة
# ... إجراء التغييرات ...
git add .              # إعداد جميع التغييرات
git commit -m "fix: increase timeout to 30s"
git push origin fix/timeout-config   # دفع إلى البعيد
# فتح طلب سحب للمراجعة
```

## استراتيجيات التفرع

كيف تنظم فريقك الفروع يؤثر بشكل كبير على مدى تكرار التكامل وسرعة وصول التغييرات إلى الإنتاج.

### GitFlow

يستخدم GitFlow عدة فروع طويلة العمر:

- **`main`** — كود جاهز للإنتاج. يُدمج فقط من فروع الإصدار أو الإصلاح العاجل.
- **`develop`** — فرع التكامل حيث تدمج فروع الميزات.
- **`feature/*`** — ميزات فردية، متفرعة من develop.
- **`release/*`** — التحضير للإصدار، متفرعة من develop.
- **`hotfix/*`** — إصلاحات عاجلة للإنتاج، متفرعة من main.

يوفر GitFlow فصلًا واضحًا بين كود التطوير والإصدار والإنتاج. ومع ذلك، الفروع الطويلة العمر المتعددة تعني أن التغييرات تستغرق وقتًا أطول للوصول إلى الإنتاج، ويمكن أن تتراكم تعارضات الدمج.

### التطوير المعتمد على الجذع (Trunk-Based Development)

يستخدم التطوير المعتمد على الجذع فرعًا رئيسيًا واحدًا (الجذع) وفروع ميزات قصيرة العمر جدًا:

- ينشئ المطورون فروع ميزات، لكنهم يدمجونها في الفرع الرئيسي خلال ساعات وليس أيام.
- أعلام الميزات تتحكم في الوظائف غير المكتملة في الإنتاج.
- تُنشأ فروع الإصدار من الفرع الرئيسي عند الحاجة، وليس كفرع طويل العمر منفصل.

هذا النهج يتوافق بشكل أفضل مع CI/CD. التغييرات تصل إلى الفرع الرئيسي بسرعة، مما يعني اختبارها ونشرها بسرعة. خطر تعارضات الدمج ينخفض بشكل كبير.

لمعظم فرق DevOps الحديثة، التطوير المعتمد على الجذع مع فروع ميزات قصيرة العمر هو النهج الموصى به.

## البنية التحتية كمستودع

المبدأ بسيط: **كل شيء يحدد بنيتك التحتية يجب أن يكون في Git.** يشمل ذلك:

- **ملفات التكوين** — إعدادات الخوادم، nginx.conf، متغيرات البيئة (غير السرية).
- **بيانات النشر** — ملفات Kubernetes YAML، ملفات Docker Compose.
- **البنية التحتية ككود** — قوالب Terraform وPulumi وCloudFormation.
- **تعريفات خطوط أنابيب CI/CD** — `.github/workflows/`، `.gitlab-ci.yml`، `Jenkinsfile`.
- **البرامج النصية** — برامج النشر وكتيبات التشغيل ومهام الأتمتة.
- **التوثيق** — قرارات البنية التحتية وأدلة الإعداد وإجراءات الاستجابة للحوادث.

عندما يكون كل شيء في Git، تحصل على:

1. **إمكانية المراجعة** — كل تغيير يمر عبر طلب سحب. تُراجع تغييرات البنية التحتية مثل كود التطبيق.
2. **إمكانية إعادة الإنتاج** — يمكن إعادة بناء أي بيئة من المستودع. لا خوادم "فريدة" خاصة.
3. **إمكانية التدقيق** — يُظهر `git log` و `git blame` التاريخ الكامل لكل ملف.
4. **التراجع** — يتراجع `git revert` عن أي تغيير في ثوانٍ.

إذا لم يكن في Git، فهو غير موجود. إذا قام شخص ما بتغيير شيء على الخادم مباشرة، فهذا التغيير غير مرئي وغير قابل للمراجعة وسيتم الكتابة فوقه في النشر التالي.

## المراجعة المعتمدة على طلبات السحب

طلبات السحب (أو طلبات الدمج في GitLab) هي آلية مراجعة التغييرات قبل وصولها إلى الفرع الرئيسي:

1. **إنشاء فرع** — اعزل عملك في فرع ميزة.
2. **الدفع وفتح طلب سحب** — صِف ما غيّرته ولماذا.
3. **تشغيل CI تلقائيًا** — خط الأنابيب يبني ويختبر ويفحص تغييراتك.
4. **مراجعة الفريق** — يفحص المراجعون الفرق ويطرحون أسئلة ويقترحون تحسينات.
5. **معالجة التعليقات** — ادفع التزامات متابعة أو أجرِ التغييرات المطلوبة.
6. **الدمج** — بعد الموافقة ونجاح CI، يُدمج طلب السحب في الفرع الرئيسي.

لتغييرات البنية التحتية، عملية مراجعة طلبات السحب مهمة بشكل خاص. إعدادات مهلة خاطئة أو منفذ غير صحيح أو مجموعة أمان misplaced يمكن أن تُعطل الإنتاج. المراجعة تلتقط هذه المشاكل قبل وصولها إلى الأنظمة الحية.

## اصطلاحات الالتزام

رسائل الالتزام الجيدة تجعل سلسلة التدقيق مفيدة. الالتزامات التقليدية توفر تنسيقًا منظمًا:

```
type(scope): description

[optional body]

[optional footer]
```

الأنواع الشائعة:

- **feat** — ميزة جديدة (`feat(monitoring): add Prometheus scrape config`)
- **fix** — إصلاح خطأ (`fix(nginx): increase worker_connections to 4096`)
- **docs** — تغييرات التوثيق (`docs(runbook): add postgres recovery steps`)
- **chore** — مهام الصيانة (`chore(deps): update Terraform provider to v4.0`)
- **refactor** — إعادة هيكلة الكود بدون تغيير السلوك

الإصدار الدلالي يعمل جنبًا إلى جنب مع الالتزامات التقليدية. كل نوع التزام يربط بزيادة في الإصدار: `feat` يزيد الإصدار الفرعي، `fix` يزيد إصدار التصحيح، و `BREAKING CHANGE` يزيد الإصدار الرئيسي.

## نظرة على GitOps

**GitOps** يأخذ مبدأ "البنية التحتية كمستودع" إلى نهايته المنطقية: Git هو المصدر الوحيد للحقيقة للحالة المطلوبة لنظامك بالكامل.

كيف يعمل:

1. **تصريحي** — تصف الحالة المطلوبة للبنية التحتية في Git (بيانات Kubernetes، قيم Helm، تكوينات Terraform).
2. **مُصدَّر** — كل تغيير هو التزام Git بتاريخ كامل.
3. **مزامنة آلية** — حلقة تسوية (ArgoCD أو Flux) تراقب مستودع Git وتزامن الحالة الفعلية باستمرار لتطابق الحالة المطلوبة.

الفكرة الأساسية: **أنت لا تجري تغييرات مباشرة على الكتلة أبدًا.** إذا كنت تريد تغيير الإنتاج، غيّر Git. الأتمتة تتعامل مع الباقي.

إذا قام شخص ما بتغيير يدوي على الكتلة، تكتشف حلقة التسوية الانحراف وتنبهك أو تعيده تلقائيًا. هذا يزيل انحراف التكوين — القاتل الصامت لموثوقية الإنتاج.

```
يدفع المطور التغيير
        │
        ▼
  مستودع Git (مصدر الحقيقة)
        │
        ▼
  ArgoCD / Flux يكتشف التغيير
        │
        ▼
  يزامن الحالة المطلوبة ← الحالة الفعلية
        │
        ▼
  الإنتاج يطابق Git
```

يوفر GitOps أقوى ضمان في العمليات: **إذا لم يكن في Git، فهو غير موجود في الإنتاج.** هذا يجعل التدقيق تافهًا والتراجع فوريًا وانحراف التكوين مستحيلًا.
