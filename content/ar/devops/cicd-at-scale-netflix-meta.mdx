# CI/CD على نطاق واسع: Netflix وMeta

Netflix تخدم 250 مليون مشترك عبر 190 دولة. Meta تشغّل Facebook وInstagram وWhatsApp لمليارات المستخدمين على أكثر من 100,000 خادم. كلاهما ينشران الكود مئات المرات يوميًا بانقطاع شبه معدوم. أنظمة CI/CD الخاصة بهما ليست مجرد خطوط — إنها منصات هندسية تجعل النشر الآمن والسريع هو الافتراضي على نطاق يمكن أن يؤثر فيه نشر سيء واحد على مئات الملايين من الأشخاص.

هذا الدرس يفحص كيف حلت Netflix وMeta مشكلة النشر على نطاق واسع، وما المبادئ التي يمكنك تطبيقها بغض النظر عن حجم شركتك.

## Netflix: Spinnaker وهندسة الفوضى

Netflix تشغّل مئات الخدمات المصغرة على مجموعات Kubernetes عبر عدة مناطق AWS. منصة النشر الخاصة بها، **Spinnaker**، بُنيت لإدارة هذا التعقيد وتُستخدم الآن من قبل آلاف الشركات حول العالم.

### خط Spinnaker

Spinnaker يعامل النشر كسير عمل من الدرجة الأولى بمراحل صريحة:

```
Git Push ──► Build ──► Bake ──► Deploy (Dev) ──► Deploy (Staging)
               │          │                          │
            Jenkins     AMI/Docker              Canary Analysis
            cluster     image creation           (Kayenta)
                                                         │
                                                  ┌──────┴──────┐
                                                  │  Promote?    │
                                                  │  (automated) │
                                                  └──────┬──────┘
                                                         │
                                                  Deploy (Production)
                                                         │
                                                  Chaos Monkey ◄── Fault injection
                                                         │
                                                  Monitoring (Atlas)
```

**مرحلة الخبز (Bake):** هذا هو مصطلح Netflix لإنشاء قطع نشر غير قابلة للتغيير. مرحلة الخبز تأخذ مخرجات البناء وتنتج AMI (لـ EC2) أو صورة Docker (لـ Kubernetes). الصورة المخبوزة **غير قابلة للتغيير** — نفس القطعة تعبر كل مرحلة من Dev إلى الإنتاج بدون تعديل.

```json
{
  "type": "bake",
  "baseLabel": "release",
  "baseOs": "ubuntu-focal",
  "cloudProviderType": "aws",
  "package": "my-service_1.2.3_all.deb",
  "amiName": "my-service-v123-20240115"
}
```

**لماذا يهم الخبز:** إذا بنيت قطعًا مختلفة لكل بيئة، فأنت لا تختبر ما تنشره. الخبز مرة واحدة والنشر في كل مكان يضمن أن القطعة التي اجتازت اختبارات التجهيز هي نفس القطعة التي تصل إلى الإنتاج.

### Kayenta: تحليل الكناري الآلي

Netflix بنت **Kayenta** لأتمتة تحليل الكناري. بدلاً من مهندسين يراقبون لوحات المعلومات، Kayenta تقارن المقاييس بين نشر الكناري والأساسي باستخدام طرق إحصائية:

```
Canary (v1.2) ──► Metrics ──► Kayenta ──► Statistical Analysis ──► Decision
                                                               │
Baseline (v1.1) ──► Metrics ──►                               │
                                                               ├── Promote (v1.2 → production)
                                                               └── Rollback (keep v1.1)
```

**ما تقارنه Kayenta:**
- **زمن الاستجابة** — أوقات استجابة p50, p95, p99
- **معدل الأخطاء** — نسبة 5xx
- **المعالج/الذاكرة** — استخدام الموارد
- **مقاييس مخصصة** — خاصة بالتطبيق (مثل وقت بدء الفيديو، معدل التخزين المؤقت)

**المنهج الإحصائي:** Kayenta لا تكتشف فقط ما إذا كانت مقاييس الكناري أسوأ. تستخدم **اختبارات Mann-Whitney U** و**اختبارات t** لتحديد ما إذا كان الفرق ذا دلالة إحصائية. هذا يمنع التراجعات الكاذبة من ضوضاء المقاييس الطبيعية بينما يلتقط التراجعات الحقيقية.

### Chaos Monkey: التحقق من المرونة

**Chaos Monkey** من Netflix يوقف حالات الإنتاج عشوائيًا خلال ساعات العمل. هذا ليس حادثًا — إنها استراتيجية متعمدة لضمان أن الخدمات يمكنها التعامل مع الأعطال بسلاسة.

**كيف يتكامل مع النشر:**

1. يُنشر إصدار جديد عبر الكناري.
2. Chaos Monkey يستهدف الحالات التي تشغل إصدار الكناري.
3. إذا تعامل الكناري مع إنهاء الحالات دون تدهور تجربة المستخدم، يُرقَّى الكناري.
4. إذا فشل الكناري تحت الفوضى، يُتراجَع عنه.

هذه **هندسة الفوضى** — حقن أعطال مضبوطة للتحقق من أن نظامك مرن. وسّعت Netflix Chaos Monkey إلى **جيش القردة** (Chaos Monkey، Latency Monkey، Conformity Monkey، إلخ) لاختبار أوضاع الفشل المختلفة.

### مراقبة Atlas

Netflix بنت **Atlas** كنظام مراقبة وتنبيه زمني داخلي. Atlas يتعامل مع ملايين نقاط البيانات في الثانية ويشغل كل تحليل الكناري ولوحات المعلومات والتنبيهات.

```
Application ──► Atlas Agent ──► Atlas Backend ──► Dashboards
                  │                                  │
                  └──► Kayenta (canary analysis)     └──► Alerts (PagerDuty)
```

## Meta: النشر المستمر على نطاق مليار مستخدم

Meta تتخذ نهجًا مختلفًا جذريًا عن Netflix. بدلاً من Kubernetes والبنية التحتية السحابية، Meta تعمل على **خوادم معدنية عارية وأجهزة افتراضية مخصصة** في مراكز بياناتها الخاصة. نظام النشر الخاص بها يدفع إصدارات ثنائية جديدة مباشرة إلى أكثر من 100,000 خادم.

### مسار نشر Meta

```
Git Push ──► Buck2 Build ──► Feature Flag Check ──► A/B Test Gate ──► Gradual Rollout
                │                  │                      │                  │
           Incremental         Gatekeeper             Statistical       FB Trench
           build system        (on/off)               significance      (region-by-region)
```

### نظام البناء Buck2

**Buck2** من Meta هو نظام بناء تزايدي مكتوب بلغة Rust أسرع بـ 100 مرة من Buck v1. الميزات الرئيسية:

- **بناء تزايدي** — يعيد بناء ما تغير فقط. تغيير سطر واحد قد يُعاد بناؤه في ثوانٍ، وليس دقائق.
- **تنفيذ موزع** — عمل البناء يُوزَّع عبر مئات الأجهزة.
- **بناء محكم** — مخرجات البناء حتمية. نفس المدخل ينتج دائمًا نفس المخرج.

```
Monorepo (millions of files)
    │
    ├── Developer changes 1 file
    │
    ├── Buck2 analyzes dependency graph
    │   (only affected targets)
    │
    ├── Rebuilds only changed targets
    │   (distributed across build farm)
    │
    └── Binary ready in seconds/minutes
        (not hours)
```

### Gatekeeper: فصل النشر عن الإصدار

**Gatekeeper** من Meta هو نظام علامات الميزات. يفصل **النشر** (الكود على الخوادم) عن **الإصدار** (الكود مرئي للمستخدمين).

```
Deploy to 100% of servers ──► Feature OFF ──► No users see new feature
                                   │
                                   ▼
                            Gradually enable ──► 1% of users ──► 5% ──► 50% ──► 100%
                                   │
                                   ▼
                            Monitor metrics at each stage
                                   │
                                   ├── Healthy? ──► Increase percentage
                                   └── Unhealthy? ──► Instant disable (no redeploy needed)
```

**لماذا هذا مهم:** Meta تنشر الكود إلى الإنتاج بشكل مستمر، لكن فرق المنتجات تتحكم متى تصبح الميزات مرئية. ميزة معطلة تُعطَّل بقلب علامة — بدون تراجع، بدون إعادة نشر، بدون انتظار. الكود موجود بالفعل على كل خادم؛ يحتاج فقط إلى التشغيل.

### بوابة اختبار A/B

قبل أن تصل أي ميزة إلى جمهور واسع، Meta تشغّل **تجارب A/B** مع فحوصات دلالة إحصائية:

1. تُفعّل الميزة لـ 1% من المستخدمين (مجموعة المعالجة).
2. تُجمع المقاييس لمجموعتي المعالجة والضبط.
3. اختبارات إحصائية تحدد ما إذا كانت الميزة تحسن أو تضر المقاييس الرئيسية.
4. فقط بعد الوصول إلى دلالة إحصائية تنتقل الميزة إلى نشر أوسع.

هذا يمنع الميزات التي تبدو جيدة في التطوير من تدهور التجربة على نطاق واسع.

### FB Trench: النشر التدريجي

**FB Trench** هو نظام النشر التدريجي من Meta. يدفع إصدارات ثنائية جديدة إلى الخوادم منطقة بمنطقة:

```
Region 1 (1%) ──► Monitor ──► Region 2 (5%) ──► Monitor ──► Region 3 (20%) ──► ...
```

في كل مرحلة، مراقبة آلية تفحص معدلات الأخطاء وزمن الاستجابة واستهلاك الموارد. إذا تدهور أي مقياس، يتوقف النشر أو يُعكس.

### ODS: إحصائيات على الجهاز

**ODS (On-Device Stats)** من Meta يوفر تقارير أخطاء فورية من مليارات أجهزة العملاء. على عكس المراقبة من جانب الخادم التي ترى فقط أخطاء الخادم، ODS يلتقط:

- أعطال وأخطاء من جانب العميل
- فشل تصيير واجهة المستخدم
- انتهاء مهلة الشبكة من منظور المستخدم
- مشاكل خاصة بالجهاز (ضغط الذاكرة، استنزاف البطارية)

هذا يعطي Meta صورة كاملة عن تجربة المستخدم، وليس فقط صحة الخادم.

## المبادئ المشتركة

رغم البنى المتباينة جذريًا، Netflix وMeta تشتركان في مبادئ نشر أساسية:

### 1. قطع غير قابلة للتغيير

كلا الشركتين تبني مرة واحدة وتنشر في كل مكان. نفس القطعة (AMI، صورة Docker، أو ملف ثنائي) التي تجتاز الاختبارات في التجهيز هي التي تُنشر في الإنتاج. لا بناء "خاص" للإنتاج.

```
Build once ──► Test ──► Deploy same artifact to dev, staging, production
```

**السبب:** إذا بنيت قطعًا مختلفة لبيئات مختلفة، فأنت لا تختبر ما تنشره. القطع غير القابلة للتغيير تقضي على مشكلة "يعمل في التجهيز، معطل في الإنتاج."

### 2. التراجع الآلي

كلا النظامين يمكنهما التراجع في ثوانٍ عند اكتشاف المشاكل. لا حاجة لاستدعاء إنسان، ولا غرفة أزمات. النظام يكتشف المشكلة ويعود تلقائيًا.

- **Netflix:** Kayenta تكتشف تدهور الكناري وتطلق تراجعًا عبر Spinnaker.
- **Meta:** FB Trench يتوقف عن النشر ويعود إلى الإصدار الثنائي السابق.

### 3. قابلية الملاحظة الكاملة

لا يمكنك النشر بأمان على نطاق واسع إذا لم تستطع رؤية ما يحدث. كلا الشركتين تستثمر بكثافة في المراقبة:

- **Netflix:** Atlas (مقاييس الخادم) + Eureka (اكتشاف الخدمات) + Edda (تتبع موارد AWS)
- **Meta:** ODS (مقاييس العميل) + مقاييس جانب الخادم + لوحات معلومات مخصصة

**المبدأ:** كل نشر يجب أن يكون قابلًا للملاحظة. إذا نشرت بدون مراقبة، فأنت تطير بلا رؤية.

## تطبيق هذه المبادئ على أي نطاق

لا تحتاج إلى Spinnaker أو نظام بناء مخصص لتطبيق هذه المبادئ:

| ممارسة Netflix/Meta | تطبيقك |
|---------------------|--------|
| خطوط Spinnaker | GitHub Actions / GitLab CI بمراحل |
| تحليل الكناري بـ Kayenta | Argo Rollouts مع تحليل المقاييس |
| Chaos Monkey | Litmus Chaos أو Chaos Mesh |
| علامات ميزات Gatekeeper | LaunchDarkly، Unleash، أو علامات مخصصة |
| بناء تزايدي بـ Buck2 | تخزين مؤقت لطبقات Docker + BuildKit |
| نشر تدريجي بـ FB Trench | كناري Kubernetes مع Flagger |
| مراقبة Atlas | Prometheus + Grafana |

## الخلاصات الرئيسية

- **اخبز مرة واحدة، انشر في كل مكان** — القطع غير القابلة للتغيير تقضي على أخطاء البيئات المحددة.
- **تحليل الكناري الآلي** (Kayenta) يستبدل مراقبة الإنسان للوحات المعلومات بالمقارنة الإحصائية.
- **هندسة الفوضى** تتحقق من المرونة بتعطيل الأشياء عمدًا في الإنتاج.
- **علامات الميزات** (Gatekeeper) تفصل النشر عن الإصدار — الكود يُشحَن دون أن يكون مرئيًا.
- **النشر التدريجي** (FB Trench) يحد نطاق الضرر بالنشر منطقة بمنطقة.
- **قابلية الملاحظة الكاملة** غير قابلة للتفاوض — لا يمكنك النشر بأمان لما لا تستطيع رؤيته.
