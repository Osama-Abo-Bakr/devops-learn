# Containers 101: Your First Step Into the Container World

Imagine you're a shipping company in the 1960s. Every piece of cargo — cars, grain, electronics — had to be loaded and unloaded differently. It was slow, expensive, and error-prone. Then the shipping container was invented: a standardized box that fits on any ship, train, or truck. No matter what's inside, the outside looks the same to the infrastructure handling it.

**Docker containers are the shipping containers of software.** They package your application along with everything it needs to run — code, runtime, libraries, environment variables, and configuration — into a single, standardized unit. That unit runs identically on your laptop, your coworker's machine, and the production server. No more "it works on my machine."

## What Is a Container, Really?

A container is a lightweight, isolated process running on your host operating system. It shares the host's kernel but has its own filesystem, network interfaces, and process space. Think of it like an apartment in a building: you share the foundation and the plumbing (the kernel), but your unit has its own walls, doors, and furniture (your app and its dependencies).

Unlike a virtual machine, a container doesn't need a full operating system installed inside it. This makes containers start in seconds instead of minutes, and weigh megabytes instead of gigabytes.

## Containers vs. Virtual Machines

This is the comparison everyone asks about, so let's make it crystal clear.

| Aspect | Containers | Virtual Machines |
|--------|-----------|-----------------|
| **Startup time** | Seconds | Minutes |
| **Size** | MBs (tens to hundreds) | GBs (several per VM) |
| **Isolation** | Process-level (shared kernel) | Hardware-level (full guest OS) |
| **Performance** | Near-native | Overhead from hypervisor |
| **OS** | Share the host kernel | Each VM has its own full OS |
| **Resource usage** | Minimal — only what the app needs | Each VM reserves fixed RAM/CPU |
| **Use case** | Microservices, CI/CD, scaling | Legacy monoliths, different OS needs |

**The key insight:** VMs virtualize hardware. Containers virtualize the operating system. That's why containers are so much lighter — they don't carry the weight of a second operating system.

**When would you pick a VM?** When you genuinely need a different kernel. If your app requires Windows and your host runs Linux, you need a VM. For most server-side applications, containers are the better choice.

## Why Does It Matter?

- **Consistency across environments** — The container that works on your laptop is the exact same container running in production. No more environment-related bugs.
- **Resource efficiency** — You can run dozens of containers on a single host where you might only fit a few VMs. This translates directly to lower cloud costs.
- **Lightning-fast startup** — Containers start in seconds, making them ideal for auto-scaling and CI/CD pipelines where speed matters.
- **Isolation by default** — Each container runs in its own sandbox. A misbehaving app in one container won't crash another.
- **Immutable infrastructure** — Containers are built from images that don't change. If something breaks, you redeploy the same image rather than patching a running server.

## Running Your First Container

Enough theory — let's get our hands dirty. We'll run an Nginx web server, one of the most popular containers on Docker Hub.

```bash
docker run -d -p 8080:80 nginx
```

Let's break down every part of this command:

- **`docker run`** — Tells Docker to create AND start a container. It's a combination of `docker create` and `docker start`.
- **`-d`** — "Detached" mode. The container runs in the background so your terminal stays free. Without this flag, the container would run in the foreground and your terminal would show Nginx's log output.
- **`-p 8080:80`** — Port mapping. The format is `HOST_PORT:CONTAINER_PORT`. Nginx listens on port 80 inside the container, but we're mapping it to port 8080 on your machine. This means you can open `http://localhost:8080` in your browser to see the Nginx welcome page.
- **`nginx`** — The name of the image to use. Docker will download it from Docker Hub if you don't have it locally.

**What happens the first time you run this?**

```
Unable to find image 'nginx:latest' locally
latest: Pulling from library/nginx
a2abf6c23d55: Pull complete
a9edb18cadd1: Pull complete
...
Status: Downloaded newer image for nginx:latest
a1b2c3d4e5f6...
```

Docker downloads the Nginx image in layers. On subsequent runs, it reuses those cached layers — so `docker run` is nearly instant.

## Exploring Your Container

Your container is running in the background. Let's learn the commands to inspect and manage it.

**See what's running:**

```bash
docker ps
```

```
CONTAINER ID   IMAGE   STATUS          PORTS                  NAMES
a1b2c3d4e5f6   nginx   Up 2 minutes    0.0.0.0:8080->80/tcp   sharp_turing
```

The container gets a random name (`sharp_turing` here) and a unique ID. You can use either to reference this container.

**View the logs:**

```bash
docker logs a1b2c3d4e5f6
```

This shows everything the container has printed to stdout and stderr. For Nginx, that means access logs and error logs. Add `-f` to follow the logs in real time (like `tail -f`).

**Jump inside the container:**

```bash
docker exec -it a1b2c3d4e5f6 sh
```

- **`exec`** — Execute a command inside a running container
- **`-i`** — Keep STDIN open (interactive)
- **`-t`** — Allocate a pseudo-TTY (gives you a proper terminal)
- **`sh`** — The shell command to run (Alpine-based images use `sh` instead of `bash`)

Once inside, you can explore the container's filesystem, check processes, and inspect the environment. Type `exit` to leave.

**See resource usage:**

```bash
docker stats a1b2c3d4e5f6
```

This shows live CPU, memory, and network usage — like `top` but for your container.

## The Container Lifecycle

Containers have a clear lifecycle. Understanding it is essential to managing them well.

```
Created  →  Running  →  Paused  →  Stopped  →  Removed
   ↓          ↓           ↓          ↓           ↓
 create     start       pause      stop         rm
            run                    kill
```

| Command | What it does | When to use it |
|---------|-------------|----------------|
| `docker create` | Creates a container without starting it | When you need to set up a container first |
| `docker start` | Starts a stopped or created container | Restarting an existing container |
| `docker run` | Creates AND starts a container | Most common — first-time launch |
| `docker pause` | Freezes all processes in the container | Temporary freeze (rarely used) |
| `docker unpause` | Resumes a paused container | After pausing |
| `docker stop` | Sends SIGTERM, then SIGKILL after timeout | Graceful shutdown |
| `docker kill` | Sends SIGKILL immediately | When stop doesn't work |
| `docker rm` | Removes a stopped container | Cleanup after you're done |

**Important distinction:** `docker stop` gives the container a chance to shut down gracefully (10 seconds by default). `docker kill` is instant and forceful. Always prefer `stop` unless the container is unresponsive.

**Clean up everything at once:**

```bash
docker container prune
```

This removes all stopped containers. Useful when your system accumulates dozens of old containers.

## Port Mapping Explained

Port mapping is how you expose a service running inside a container to the outside world. Let's make this intuitive.

Think of a container like a house with the curtains drawn. The application inside is doing its thing, but nobody outside can reach it. Port mapping is like installing a doorbell — it connects a specific port on your host machine to a specific port inside the container.

```bash
# Basic mapping: host port 8080 → container port 80
docker run -d -p 8080:80 nginx

# Map to a specific IP only (not accessible from other networks)
docker run -d -p 127.0.0.1:8080:80 nginx

# Map multiple ports
docker run -d -p 8080:80 -p 8443:443 nginx

# Random host port (Docker picks an available port)
docker run -d -p 80 nginx
# Then check the mapped port with: docker port <container_id>
```

**The `HOST:CONTAINER` pattern is critical.** The left side is your machine, the right side is the container. Mix them up and nothing works.

## Detached vs. Foreground Mode

This trips up a lot of beginners, so let's clarify it.

**Detached (`-d`):** The container runs in the background. You get your terminal back immediately. This is what you want for servers (web apps, databases, message queues).

```bash
docker run -d nginx
# Output: container ID, then your terminal is free
```

**Foreground (no `-d`):** The container runs attached to your terminal. You see all output in real time, and pressing `Ctrl+C` stops the container. This is useful for one-off tasks.

```bash
docker run nginx
# You see Nginx logs streaming in your terminal
# Ctrl+C stops the container
```

**The best of both worlds:** Run detached, then attach when you need to see logs:

```bash
docker run -d --name my-nginx nginx
docker attach my-nginx    # See live output
# Press Ctrl+C to detach without stopping (Ctrl+P, Ctrl+Q)
```

## Practical Example: Running a Development Web Server

Let's put it all together with a realistic workflow. Suppose you want to spin up a quick web server to test a static website.

**Step 1: Create a project directory**

```bash
mkdir my-website && cd my-website
echo "<h1>Hello from Docker!</h1>" > index.html
```

**Step 2: Run Nginx with your content mounted in**

```bash
docker run -d \
  --name my-dev-server \
  -p 8080:80 \
  -v $(pwd):/usr/share/nginx/html:ro \
  nginx
```

Let's break down the new flag:

- **`--name my-dev-server`** — Give the container a memorable name instead of a random one
- **`-v $(pwd):/usr/share/nginx/html:ro`** — Mount your current directory into the container's web root. The `:ro` flag means read-only — the container can't modify your files.

**Step 3: Verify it works**

Open `http://localhost:8080` in your browser. You should see "Hello from Docker!"

**Step 4: Make a change**

Edit `index.html` on your machine and refresh the browser. The changes appear instantly because the volume mount is live.

**Step 5: Clean up**

```bash
docker stop my-dev-server
docker rm my-dev-server
```

## Common Pitfalls & Tips

- **Forgetting to map ports** — If you run a web container without `-p`, it's running but inaccessible. You won't see any errors; it just silently doesn't work. Always check your port mappings.

- **Using `latest` tag in production** — The `latest` tag is a moving target. `nginx:latest` today might be different from `nginx:latest` next month. Pin your versions: `nginx:1.25-alpine`.

- **Not cleaning up** — Every `docker run` creates a container. Even stopped containers consume disk space. Get in the habit of adding `--rm` for temporary containers: `docker run --rm nginx` automatically removes the container when it stops.

- **Confusing images and containers** — An image is the blueprint. A container is a running instance of that image. You can run many containers from the same image. Think of it like classes and objects in programming.

- **Container data disappears on removal** — If you write data inside a container and then remove it, that data is gone. We'll cover volumes in a later lesson, but keep this in mind now.

- **`docker stop` hanging** — If your container doesn't handle SIGTERM properly, `docker stop` will hang for 10 seconds before killing it. Use `docker kill` as a last resort, but the real fix is making your app handle shutdown signals.
