# ConfigMaps and Secrets

Hardcoding configuration in container images is a bad practice. ConfigMaps and Secrets let you inject configuration at runtime, keeping images reusable across environments.

## ConfigMaps

ConfigMaps store non-sensitive configuration — database hosts, feature flags, log levels.

### Create from literal values

```bash
kubectl create configmap app-config \
  --from-literal=DB_HOST=db.production \
  --from-literal=LOG_LEVEL=info \
  --from-literal=MAX_CONNECTIONS=100
```

### Create from a file

```bash
kubectl create configmap nginx-config \
  --from-file=nginx.conf
```

### Use as environment variables

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:1.0
      envFrom:
        - configMapRef:
            name: app-config
```

### Use as mounted files

```yaml
volumes:
  - name: config
    configMap:
      name: nginx-config
volumeMounts:
  - name: config
    mountPath: /etc/nginx/nginx.conf
    subPath: nginx.conf
```

## Secrets

Secrets are similar to ConfigMaps but designed for sensitive data. They are base64-encoded (not encrypted by default — enable encryption at rest for production).

### Create a secret

```bash
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=s3cur3P@ss
```

### Use as environment variables

```yaml
env:
  - name: DB_USER
    valueFrom:
      secretKeyRef:
        name: db-credentials
        key: username
  - name: DB_PASS
    valueFrom:
      secretKeyRef:
        name: db-credentials
        key: password
```

## ConfigMap vs Secret

| Feature | ConfigMap | Secret |
|---------|-----------|--------|
| Storage | Plain text | Base64 encoded |
| Size limit | 1 MB | 1 MB |
| Use for | URLs, flags, config | Passwords, keys, tokens |
| Access control | Namespace-scoped | Namespace-scoped + RBAC |
| Encryption | No | Optional (encryption at rest) |

## Best Practices

- Use ConfigMaps for non-sensitive config
- Use Secrets for sensitive data
- Don't put Secrets in Git — use Sealed Secrets or external secret managers
- Reference individual keys (`secretKeyRef`) instead of `envFrom` for least privilege
- Enable encryption at rest for production clusters
