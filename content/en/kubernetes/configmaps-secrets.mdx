# ConfigMaps and Secrets

Imagine you're packing for a trip. You wouldn't sew your destination into your jacket -- you'd write it on a tag you can swap out. That's exactly the problem ConfigMaps and Secrets solve in Kubernetes. **Configuration should be separate from your container image** so the same image can run in development, staging, and production with different settings.

Without this separation, you'd need a different Docker image for every environment. Change the database URL? Rebuild the image. Change the log level? Rebuild again. That's wasteful, slow, and error-prone. ConfigMaps and Secrets let you inject configuration at runtime, keeping your images clean, reusable, and portable.

## The Configuration Problem

Here's what happens when you bake configuration into images:

- **One image per environment.** Dev, staging, and prod each need their own image with hardcoded values.
- **A simple config change requires a full rebuild.** Changing a log level shouldn't mean rebuilding and redeploying a 500MB image.
- **Secrets in images are dangerous.** Database passwords baked into images end up in your container registry. Anyone with registry access has your credentials.
- **No flexibility.** Can't override settings without rebuilding. Can't experiment with different configs quickly.

The solution: **externalize everything that changes between environments.** ConfigMaps handle non-sensitive settings. Secrets handle sensitive data.

## What is a ConfigMap?

A **ConfigMap** is a Kubernetes object that stores non-sensitive configuration data as key-value pairs. Your application reads this data at runtime through environment variables or mounted files.

Think of a ConfigMap like a settings menu on your phone. You don't rewrite the phone's operating system when you want to change the wallpaper -- you update a setting. ConfigMaps do the same thing for your containers.

### What Goes in a ConfigMap?

- Database hostnames and port numbers
- API URLs for external services
- Feature flags (enable/disable features)
- Log levels (`info`, `debug`, `warning`)
- Application timeouts and retry settings
- Any non-sensitive configuration value

### What Does NOT Go in a ConfigMap?

- Database passwords (use a **Secret**)
- API keys and tokens (use a **Secret**)
- TLS private keys (use a **Secret**)
- Any data you wouldn't paste into a public Slack channel

## Why Does It Matter?

- **Same image, multiple environments.** Build once, deploy everywhere. ConfigMaps provide the environment-specific values at runtime.
- **Fast configuration changes.** Update a ConfigMap and restart your Pods -- no rebuild needed. Change a log level in seconds, not minutes.
- **Decoupled architecture.** Your application code doesn't need to know where it's running. It just reads config from its environment.
- **Centralized configuration management.** All your app's settings live in one place, version-controlled in YAML, not scattered across Dockerfiles.
- **12-factor app compliance.** ConfigMaps help you follow the "Store config in the environment" principle from the 12-factor methodology.

## Creating ConfigMaps

There are three ways to create a ConfigMap. Let's walk through each one.

### Method 1: From Literal Values

The quickest way -- specify key-value pairs directly on the command line:

```bash
kubectl create configmap app-config \
  --from-literal=DATABASE_HOST=db.production.example.com \
  --from-literal=DATABASE_PORT=5432 \
  --from-literal=LOG_LEVEL=info \
  --from-literal=MAX_CONNECTIONS=100 \
  --from-literal=CACHE_ENABLED=true
```

```text
configmap/app-config created
```

This creates a ConfigMap with five keys. It's great for quick experiments and simple configs, but becomes unwieldy for large configurations.

### Method 2: From a File

Got a config file already? Load it directly:

```bash
# Creates a ConfigMap with a key named "nginx.conf" containing the file contents
kubectl create configmap nginx-config \
  --from-file=nginx.conf
```

You can also specify a custom key name:

```bash
kubectl create configmap nginx-config \
  --from-file=my-nginx.conf=nginx.conf
```

This is perfect for entire configuration files -- Nginx configs, application.properties, JSON settings, etc.

### Method 3: From a YAML Manifest (Recommended for Production)

For version control and reproducibility, always use a YAML manifest:

```yaml
apiVersion: v1                          # Core API group
kind: ConfigMap
metadata:
  name: app-config                      # Name of the ConfigMap
  namespace: production                # Optional -- defaults to "default"
data:                                   # The actual configuration data
  DATABASE_HOST: "db.production.example.com"   # Key-value pairs
  DATABASE_PORT: "5432"                        # All values are strings in ConfigMaps
  LOG_LEVEL: "info"
  MAX_CONNECTIONS: "100"
  CACHE_ENABLED: "true"
  APP_TIMEOUT: "30"                            # Even numbers are stored as strings
```

**Important:** All values in ConfigMaps are stored as strings. Even if you write `5432` without quotes, Kubernetes treats it as a string. Keep this in mind when your application expects a specific type.

Apply and verify:

```bash
kubectl apply -f app-config.yaml
kubectl get configmap app-config
```

```text
NAME         DATA   AGE
app-config   6      5s
```

The `DATA` column shows how many keys the ConfigMap contains.

```bash
kubectl describe configmap app-config
```

```text
Name:         app-config
Namespace:    default
Data
====
DATABASE_HOST:  db.production.example.com
DATABASE_PORT:  5432
LOG_LEVEL:      info
MAX_CONNECTIONS:100
CACHE_ENABLED:  true
APP_TIMEOUT:    30
```

## Using ConfigMaps in Pods

There are two ways to inject ConfigMap data into your containers: **environment variables** and **mounted volumes**. Each has different trade-offs.

### Option 1: Environment Variables (All Keys)

The simplest approach -- inject every key in the ConfigMap as an environment variable:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:1.0
      envFrom:                                # Inject ALL keys from the ConfigMap
        - configMapRef:
            name: app-config                  # Reference the ConfigMap by name
```

Every key in `app-config` becomes an environment variable inside the container. Your app reads `DATABASE_HOST`, `LOG_LEVEL`, etc. just like any other environment variable.

**Pros:** Simple, all config in one shot.
**Cons:** Every key becomes an env var, even ones you might not want. No granular control.

### Option 2: Environment Variables (Specific Keys)

For more control, reference individual keys:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:1.0
      env:                                    # Define individual environment variables
        - name: DB_HOST                       # The env var name inside the container
          valueFrom:
            configMapKeyRef:
              name: app-config                # Which ConfigMap to read from
              key: DATABASE_HOST              # Which key in the ConfigMap
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
```

**Pros:** Only expose the keys you need. You can rename keys (e.g., `DATABASE_HOST` becomes `DB_HOST`).
**Cons:** More verbose YAML. Need to list each key individually.

### Option 3: Mounted Volumes

For configuration files (like `nginx.conf` or `application.yml`), mount them as files:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-app
spec:
  containers:
    - name: nginx
      image: nginx:alpine
      volumeMounts:                           # Mount the volume into the container
        - name: config-volume                 # Reference the volume defined below
          mountPath: /etc/nginx/nginx.conf     # Where to mount it inside the container
          subPath: nginx.conf                 # Mount only this specific key (not the whole directory)
          readOnly: true                      # Config files should be read-only
  volumes:                                    # Define the volumes available to the Pod
    - name: config-volume                     # Name must match the volumeMount reference
      configMap:
        name: nginx-config                    # Which ConfigMap to mount
```

Without `subPath`, the entire ConfigMap is mounted as a directory, with each key becoming a file. With `subPath`, you mount a single key as a single file. Use `subPath` when you want to replace one specific file, not overlay a whole directory.

**Key difference from environment variables:** When a ConfigMap is mounted as a volume, **updates to the ConfigMap are automatically reflected in the mounted files** (after a few seconds). Environment variables are set at Pod creation time and don't update automatically.

## What is a Secret?

A **Secret** is like a ConfigMap for sensitive data. It stores passwords, API keys, TLS certificates, and other credentials. The key difference: Secrets are **base64-encoded** and can be **encrypted at rest**.

### The Base64 Encoding Warning

This trips up almost every beginner: **base64 is NOT encryption.** It's just an encoding scheme. Anyone can decode a base64 string:

```bash
# This is NOT secure:
echo -n "s3cur3P@ss" | base64
# Output: czNjdXIzUEBzcw==

# Anyone can decode it:
echo "czNjdXIzUEBzcw==" | base64 --decode
# Output: s3cur3P@ss
```

Base64 encoding exists because Kubernetes stores data as bytes and needs a way to represent binary data (like TLS certs) alongside text. The "security" of Secrets comes from:

1. **RBAC** -- You can restrict who can read Secrets (unlike ConfigMaps)
2. **Encryption at rest** -- Enable this in your cluster's API server config
3. **Audit logging** -- Access to Secrets is logged
4. **External secret managers** -- Integrate with AWS Secrets Manager, HashiCorp Vault, etc.

## Creating Secrets

### Method 1: From Literal Values

```bash
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=s3cur3P@ss
```

Note: `kubectl` handles the base64 encoding for you. You pass the raw value, and it encodes it automatically.

### Method 2: From Files

```bash
kubectl create secret tls app-tls \
  --from-file=tls.crt=server.crt \
  --from-file=tls.key=server.key
```

This creates a TLS-type Secret specifically for certificates.

### Method 3: From a YAML Manifest

When writing YAML directly, you **must** base64-encode the values yourself:

```bash
# Encode the values first
echo -n "admin" | base64
# Output: YWRtaW4=

echo -n "s3cur3P@ss" | base64
# Output: czNjdXIzUEBzcw==
```

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque                            # Opaque = arbitrary key-value data
data:                                   # Values MUST be base64-encoded
  username: YWRtaW4=                    # base64("admin")
  password: czNjdXIzUEBzcw==            # base64("s3cur3P@ss")
```

You can also use `stringData` instead of `data` to avoid manual encoding:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:                             # Raw strings -- Kubernetes encodes them for you
  username: admin
  password: s3cur3P@ss
```

`stringData` is write-only. When you read the Secret back with `kubectl get`, the values appear base64-encoded under `data`.

## Using Secrets in Pods

### As Environment Variables (Individual Keys)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:1.0
      env:
        - name: DB_USER                     # Environment variable name
          valueFrom:
            secretKeyRef:                     # Read from a Secret (not ConfigMap)
              name: db-credentials            # Which Secret
              key: username                   # Which key
        - name: DB_PASS
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
```

**Best practice:** Use `secretKeyRef` for individual keys instead of `envFrom` with `secretRef`. This follows the principle of least privilege -- your app only gets the specific secrets it needs, not all of them.

### As Mounted Files

```yaml
volumes:
  - name: secret-volume
    secret:
      secretName: db-credentials             # Mount all keys as files
      defaultMode: 0400                      # Read-only for owner only (octal permission)
volumeMounts:
  - name: secret-volume
    mountPath: /etc/secrets                  # Each key becomes a file: /etc/secrets/username, /etc/secrets/password
    readOnly: true
```

Inside the container, your app reads `/etc/secrets/username` and `/etc/secrets/password` as regular files. The files contain the **decoded** values (not base64).

## ConfigMap vs Secret: Comparison

| Feature | ConfigMap | Secret |
|---------|-----------|--------|
| **Purpose** | Non-sensitive configuration | Sensitive data (passwords, keys, certs) |
| **Storage** | Plain text in etcd | Base64-encoded in etcd |
| **Size limit** | 1 MiB per ConfigMap | 1 MiB per Secret |
| **RBAC** | Standard namespace access | More restrictive access by default |
| **Encryption at rest** | No | Optional (must enable in API server config) |
| **Memory handling** | Stored in disk-based filesystem | Can be backed by tmpfs (RAM) for extra security |
| **Creation from literals** | Yes | Yes |
| **Creation from files** | Yes | Yes (including TLS type) |
| **Update propagation** | Volume mounts: auto | Volume mounts: auto |
| **Git-friendly** | Yes -- safe to commit | No -- avoid committing raw Secrets |

## Practical Example: Configurable Web Application

Let's deploy a web application that reads its database connection and log settings from a ConfigMap, and its database credentials from a Secret.

**Project structure:**

```text
k8s/
├── configmap.yaml
├── secret.yaml
├── deployment.yaml
└── service.yaml
```

**configmap.yaml:**

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: web-app-config
data:
  DATABASE_HOST: "postgres.production.svc.cluster.local"   # Internal DNS for the database
  DATABASE_PORT: "5432"                                    # PostgreSQL default port
  DATABASE_NAME: "app_production"                          # Database name
  LOG_LEVEL: "info"                                        # Logging verbosity
  CACHE_TTL: "300"                                         # Cache time-to-live in seconds
  MAX_UPLOAD_SIZE: "10MB"                                  # Maximum file upload size
```

**secret.yaml:**

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: web-app-secrets
type: Opaque
stringData:                                # Using stringData to avoid manual base64
  DATABASE_USER: "app_user"
  DATABASE_PASSWORD: "Ch@ng3M3!NPr0d"
  API_KEY: "sk-prod-abc123xyz789"
```

**deployment.yaml:**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
        - name: web-app
          image: my-registry/web-app:2.0.0
          ports:
            - containerPort: 3000
          envFrom:
            - configMapRef:
                name: web-app-config        # All ConfigMap keys become env vars
          env:
            - name: DATABASE_USER          # Inject specific secret keys
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: DATABASE_USER
            - name: DATABASE_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: DATABASE_PASSWORD
            - name: API_KEY
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: API_KEY
          resources:
            requests:
              cpu: "200m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
```

Deploy everything:

```bash
kubectl apply -f k8s/
kubectl get pods -l app=web-app
```

```text
NAME                       READY   STATUS    RESTARTS   AGE
web-app-7d8b9c5f4-2xzkm   1/1     Running   0          30s
web-app-7d8b9c5f4-5njpw   1/1     Running   0          30s
web-app-7d8b9c5f4-8qrty   1/1     Running   0          30s
```

Verify the environment variables inside a Pod:

```bash
kubectl exec -it web-app-7d8b9c5f4-2xzkm -- env | grep -E "DATABASE|LOG_LEVEL|API_KEY"
```

```text
DATABASE_HOST=postgres.production.svc.cluster.local
DATABASE_PORT=5432
DATABASE_NAME=app_production
DATABASE_USER=app_user
DATABASE_PASSWORD=Ch@ng3M3!NPr0d
LOG_LEVEL=info
API_KEY=sk-prod-abc123xyz789
```

Your app sees these as regular environment variables. It has no idea they came from Kubernetes -- which is exactly the point.

## Common Pitfalls and Tips

| Pitfall | What Goes Wrong | How to Fix It |
|---------|----------------|---------------|
| **Baking config into images** | Can't change settings without rebuilding | Use ConfigMaps and Secrets for all environment-specific values |
| **Storing Secrets in Git** | Passwords and keys end up in version control | Use Sealed Secrets, External Secrets Operator, or SOPS to encrypt before committing |
| **Forgetting base64 in YAML** | Secret data field gets invalid input | Use `stringData` for raw values, or remember to encode with `echo -n "value" \| base64` |
| **Assuming base64 = encryption** | Secrets are trivially decoded by anyone who can read them | Enable encryption at rest; use RBAC to restrict access |
| **Not setting resource limits** | ConfigMap/Secret mounted as volume can consume memory | Set `readOnly: true` on volume mounts |
| **Updating ConfigMaps without restarting Pods** | Env var changes aren't picked up by running containers | Volume-mounted ConfigMaps auto-update; env vars require a Pod restart |
| **Using the same ConfigMap across namespaces** | ConfigMaps are namespace-scoped -- you can't reference across namespaces | Create a ConfigMap in each namespace, or use tools that sync ConfigMaps |
| **Putting large files in ConfigMaps** | Hits the 1 MiB limit | Use PersistentVolumes for large files; ConfigMaps are for small config values |

### Pro Tips

- Use `kubectl create configmap --from-env-file=.env` to convert a `.env` file into a ConfigMap.
- Use **Sealed Secrets** (`kubeseal`) if you need to store encrypted Secrets in Git safely.
- For enterprise setups, consider the **External Secrets Operator** to sync secrets from AWS Secrets Manager, HashiCorp Vault, or Azure Key Vault.
- Always reference specific secret keys (`secretKeyRef`) rather than using `envFrom` with secrets -- this limits exposure if the Secret contains more data than your app needs.
- Enable **encryption at rest** in your API server config (`--encryption-provider-config`) for production clusters.
- Use `kubectl get secret db-credentials -o jsonpath='{.data.password}' | base64 --decode` to quickly read a Secret value when debugging.
